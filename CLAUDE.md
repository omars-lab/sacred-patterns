# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Sacred Patterns is a TypeScript library that generates sacred geometric patterns as SVG using D3.js. It bundles into a UMD module (`sacredPatterns`) viewable in a browser.

## Engineering Tenets

These are the tenets the code you produce here will be reviewed against. The reviewer (Omar) reads diffs with these in mind — when you violate one, expect to be asked to revise. Each is named for the failure mode it prevents, not the platitude it sounds like.

**These tenets apply at write time AND at read time.** When debugging, reviewing, or otherwise reading code surfaces a violation (a silent-failure path that wasted investigation time, a load-bearing function with no intent comment, a parallel structure that could be collapsed), the in-flight PR is the cheapest place to fix it. The cost of the cleanup is paid once; the cost of leaving it is paid by every future reader.

1. **Simplicity over complexity — simplify while amplifying.** Break problems into the smallest chunks that still ship value, build one chunk at a time, and prefer the smaller solution that does the job. Don't solve the whole problem in one move. "Simplifying while amplifying" means the chunk you ship should both reduce surface area AND increase what the system can do — not just one or the other. *Failure mode this prevents:* large speculative rewrites, premature abstractions, multi-PR designs delivered as one PR.
2. **Fix the root cause, don't add a fallback.** When a tool/path is broken, root-cause it and fix it. Do not add a fallback that papers over the failure. Fallbacks remove the pressure to keep the real path healthy and hide the very signal that surfaced the bug. The synthesized-warnings revert in commit d5c7c23 is the canonical example. *Failure mode this prevents:* defensive code that masks regressions; "graceful degradation" that quietly rots into the primary path.
3. **Surface, don't hide.** When something is broken, missing, or unavailable, make it loud — a blocker, an error, an explicit non-zero exit, an entry in the user-facing summary. Silent degradation (an empty array, a `None`, a no-op) is worse than a noisy failure because it looks identical to "everything is fine." *Failure mode this prevents:* the bikar-medallion-10 cascade — three iterations stranded because `overall.warnings = []` looked like "no warnings" instead of "the warnings tool is broken."
4. **Verify before claiming done.** Run the tool, read the output, confirm the assertion before saying "shipped" or "fixed." Trust no summary — including your own — that wasn't checked against the actual artifact. *Failure mode this prevents:* describing "3 backlog adds" when the diff was a 154-line rewrite, marking a task complete when the code is partial, claiming a build works because it compiled (rather than because you ran it).
5. **Read the existing shape before adding new shape.** Before introducing a new function, type, file, or data path, check whether the data, helper, or pattern already exists. Reach for what's there before building parallel structure. *Failure mode this prevents:* duplicating logic, inventing schemas the consumer can't read, building a "fallback path" when the data was already accessible (see tenet 2).
6. **Trust but verify — inherited claims are snapshots, not facts.** Hand-off docs, retrospectives, prior commit messages, prior agent summaries, and roadmap rationales describe what was true *when they were written*. They go stale silently. Before acting on an inherited claim, especially when it tells you to skip something, deviate from a default, or apply a non-obvious procedure, check the underlying artifact (the validation.json, the git diff, the actual file) and confirm the claim still holds. If it doesn't, surface the divergence and decide fresh — don't follow the stale claim *or* silently override it. *Failure mode this prevents:* the iter-13 hand-off case — `iteration-status.md` said "don't start from iter 13, that ring-addition made things worse" based on pre-warnings-array reasoning, but the freshly-validated v0.1.1 data showed iter 13's composite was actually +0.08 over iter 12. Following the doc literally would have skipped the better starting point and discarded a real signal. *This is distinct from tenet 4 (verify your own claims) — this is about claims you inherited.*
7. **Don't tune constants to make a test green — separate the algorithm from its calibration, and stop when you start going in circles.** When a test fails on a single fixture, the first reflex must NOT be to widen a tolerance, soften a confidence formula, or otherwise adjust constants until the bar moves under the data. That's *goal-seeking*: each tweak feels locally justified ("the geometry is clearly within tolerance"), but the cumulative pattern is "tune until green," which overfits the algorithm to one input and silently weakens it everywhere else. The same training set cannot both *calibrate* the constants and *judge* the algorithm — that's the ML overfitting trap, restated for geometric/heuristic code. **Stop rule:** if you've tweaked the same constant or formula twice in service of one failing case, stop. Either (a) the test is exercising a known-deferred case (mark it `xfail` or `skip` with a doc reference and move on), or (b) the spec is wrong for a structural reason that needs a separate write-up before any constant changes. Constants only change *after* you have at least two independent fixtures (one to fit, one to judge) AND a written rationale that names the noise floor being absorbed. *Failure mode this prevents:* the F3 arrangement-classifier session (2026-05-04) where I bumped `ROTATION_RADIUS_TOL_FRAC` 0.02→0.05, then `ROTATION_ANGLE_TOL_DEG` 2.0→3.0, then softened the confidence formula `1 - max_dev/tol` → `1 - 0.15·max_dev/tol`, all to make one iter-14 satellite test pass — three "local fixes" in a row, each justified in isolation, that together rewrote the spec's algorithm to fit one fixture instead of spec-and-data treating each other as independent witnesses. The right move was to mark that test `xfail` (it touches FM2 composite-arrangement territory) and surface the divergence as one decision, not three.
8. **Solve the general problem, not the specific instance.** Every failing case is a *witness* to a class of problems, not a problem in itself. We may *start* from one instance to make the class concrete (a specific bikar pattern, a specific iteration), but the deliverable is always a solution that works for every member of the class — every bikar-emitted pattern, every iteration, every reasonable input. **Stop rule before shipping any fix:** name the class the instance belongs to, name two more members of that class that the fix must also handle, and check the fix on at least one of them. If the fix only works on the originating instance, it's not a fix — it's a special case dressed up as one. *Failure mode this prevents:* a tolerance bump calibrated against iter-14's specific 3.2% radial drift that says nothing about iter-12, medallion-6, or star-7. The right framing is "what tolerance policy works for any pattern bikar's gt-emitter could produce?" — which forces investigating bikar's emission semantics (canonical vs drifting positions) instead of fitting numbers to one drift value. Every spec, detector, and threshold in the codebase should be readable as "this works for any X bikar produces" — if you can't say that, you're shipping a special case.
9. **Document intent — name the question the code answers.** Code shows *what* it does; comments must say *why this code, why now, what question is it answering*. This does NOT override the default "no comments" rule for obvious operations — it adds a higher bar: any non-trivial function, module, or block of geometric/algorithmic logic gets a short header (1–4 lines) that names the question being answered, the constraint that forced this approach, or the invariant it preserves. The reader (future-you or a future agent) should be able to tell from the header whether a proposed simplification would *change the answer to the question* — which is the only safe way to refactor load-bearing logic. **Stop rule:** before shipping any non-trivial function or module, ask "if I deleted my header comment, could a fresh reader still tell which of several plausible algorithms this is, and why we picked it?" If no, write the header. **Anti-pattern:** WHAT-comments that restate the code (`// loop over shapes`), TODO-comments without an owner or condition for resolution, and ceremonial docstrings that paraphrase the function name. *Failure mode this prevents:* the iter-3/4/5 fused-distance work is the canonical example — `derive_fused_partition_v2` is correct, but no comment in the file says "this exists because A-clustering and B-clustering disagree on lumped multi-class faces, and we want a partition that respects both views as a kernel-fusion problem (Kumar/Rai/Daumé 2011)." Without that question stated in the code, a future agent could "simplify" by removing one of the two fusion arms — and only discover at iter-N+5 that they answered a different question. The decision doc captures the intent; the *code* must too, because the code is what gets re-read on every change.
10. **Functional programming paradigm — immutable data, pure functions, composition over inheritance.** Three sub-rules, all required: (a) **immutable data:** geometric transforms and data manipulations return new instances rather than mutating inputs in place; this matches the existing convention (`rotate()`, `adjacent()`, `scaleLine()`) and extends it to every new helper. (b) **pure functions:** public APIs are single-purpose pure functions of their inputs — no hidden global state, no type-dispatching overloads (`compute_diff_images(image, image)` over `compute_diff(input)` that branches on suffix), no side effects beyond the explicit return value. Pure-functional internals are encouraged but not required when they'd hurt readability (e.g., a tight numerical inner loop). (c) **composition over inheritance:** prefer combining small functions/objects via composition rather than building deep class hierarchies; subclassing is only justified when the relationship is genuinely "is-a" *and* the subclass adds behavior not expressible as a parameter or a wrapped delegate. *Failure mode this prevents:* (a) prevents the iter-loop bug class where step N+1 inherits state from step N because a transform mutated its input — every iteration loop in this codebase assumes idempotent transforms, and one in-place mutation breaks that assumption silently. (b) prevents the qiyas 8.4 anti-pattern where `compute_diff` was overloaded on input type and consumers had to guess which behavior they were getting. (c) prevents the speculative class-hierarchy growth that tenet 1 (simplicity) warns against — every new "abstract base class" is an architectural commitment that's hard to undo, where a small set of composable functions would have served the same need with less surface area. **When you need to violate a sub-rule** (e.g., a stateful generator for a streaming pipeline, or an inheritance hierarchy that genuinely models the domain), name the violation and the reason in a comment per tenet 9.
11. **One tool path per question — patterns are inputs to the same pipeline, not special cases.** Every pattern (medallion-10, Star-10, petal-N-2ring, hexagram-lens, every future construction) is scored, validated, encoded, and rolled up through the **same code path** with the **same tools**. No per-pattern scoring drivers, no schema-version-branching detectors, no "Phase 1.A path" alongside a "Phase 1.B path." When a corpus widens, the pipeline widens once, not per-corpus. When a detector spec changes, every consumer of that spec moves together — no left-behind drivers measuring the old contract. **Stop rule before shipping any analysis result:** name which tool path produced it and confirm that *every other pattern in the corpus* was scored through the identical path with the identical knobs. If the answer is "this pattern uses driver X and the others use driver Y," stop — unify before measuring. *Failure mode this prevents:* the iter-1-to-iter-15 drift class — `validate_detector` shipped in iter-1 measuring macro identity-fidelity via canonical+shifted-grid matching, the spec moved through iters 2-15 (turning function, fused distance, α-resolver, γ-narrow) via parallel ARI cross-check scripts that *only* measured per-construction ARI, and the macro-lift rollup contract was silently abandoned. Thirteen iterations of detector improvements had no macro-fidelity signal because the unified path didn't survive contact with rapid spec change. Same defect class as tenet 2 (fallbacks paper over root cause) but specifically about *tooling fragmentation*: parallel drivers that measure different things create the illusion that everything is being measured, while the original contract rots. **Application:** before adding a new pattern to a corpus, the pipeline that scores it must be the same one that scores every existing entry; before changing a detector, every driver downstream must move with the spec or be retired in the same shipment.
12. **Commit autonomously on meaningful, tested chunks — don't ask permission per commit.** When a chunk of work is (a) meaningful (a coherent capability, fix, or migration unit — not a half-finished edit), (b) tested (its tests exist and pass; or for non-test artifacts like docs/configs, the artifact has been verified), and (c) in a clean state (no orphaned scaffolding, no debug code), commit it. Do not pause to ask "should I commit this?" — the answer is yes by default. Multiple meaningful chunks → multiple commits, not one mega-commit at the end. **Stop rule:** if you're holding tested, meaningful work uncommitted because you're "waiting for the right moment," you've already missed it — commit now. Conversely, do not commit half-finished work just to clear the working tree; the rule is "meaningful AND tested," not "any change." Commit messages still explain the why, tenet 9-style. *Failure mode this prevents:* friction-driven mega-commits that bundle three unrelated capabilities into one diff because each was "waiting for approval"; uncommitted WIP that becomes a hand-off liability when the session ends; ambiguity about which changes have been validated vs which are speculative. **Exceptions that still require explicit user approval** (per the "Executing actions with care" rules earlier in this file): pushing to remotes, force-pushing, destructive history rewrites (`reset --hard`, `rebase` on shared branches), creating or merging PRs, anything visible to others.
13. **Time can be sacrificed for quality — default to correct over quick.** When faced with a quick-and-dirty option (a tolerance bump, a parallel script, a half-migration that "we'll clean up later") versus a slower-and-correct option (root-cause fix, unified pipeline, full migration in one shipment), default to correct. The cost of a half-baked artifact compounds across every downstream consumer — every future agent who reads it, every iteration that builds on it, every "why does this look weird?" investigation. The cost of one extra hour (or day, or week) of doing it right is paid once. **Application:** if your plan has the shape "ship the workaround now, fix it properly in a follow-up," stop and ask whether the follow-up is genuinely going to happen — if it's a known-deferral pattern (tenet 2, tenet 7), the workaround is the new permanent state. Quality-over-speed is also the explicit license to spend time on reflection logs, decision docs, parity checks, and `pattern.identity.json`-style ground truths even when they feel like overhead — they're the artifacts that make the *next* iteration cheap. *Failure mode this prevents:* the cumulative-debt pattern where ten "small shortcuts" become an unreadable codebase that needs a rewrite; the iter-13 hand-off case (tenet 6) where a pre-warnings-array rationale became permanent guidance because no one paid the hour to re-validate it; the temptation to ship a tolerance bump (tenet 7) instead of authoring the decision doc that names the class.

14. **Docstrings for important functions show minimal input/output examples — fixtures by reference, not by paraphrase.** Type signatures tell you what the function accepts; examples tell you what it *means*. For any function that is load-bearing — public APIs, geometric/algorithmic logic, detectors, scoring rollups, DSL primitives, anything a future agent will need to call without reading the implementation — the docstring includes at least one concrete input/output example. **What "concrete" means:** real values (`Point(3, 4)`, not "a Point"), enough fields to disambiguate variant shapes, and the actual output (not "returns a Shape"). **Fixtures by reference:** when a canonical input lives in a fixture, pattern.gt.json, or test file, cite the path (`tests/fixtures/medallion-10/iter-14/pattern.gt.json`) rather than duplicating its contents inline — the fixture is the source of truth and prose copies rot. **Stop rule before shipping any important function:** ask "could a future agent call this correctly from the docstring alone?" If no, the docstring is incomplete — add the example. **Anti-pattern:** ceremonial `Args:`/`Returns:` blocks that paraphrase the type annotations (`points: list of Point objects representing the polygon vertices`). The type already says that; the docstring must show *what shape of input produces what shape of output*. **Companion to tenet 9 (intent comments — name the question) and tenet 15 (typed variants — show the variant in action):** intent says *why*, types say *what*, examples say *how it actually behaves on real data*. *Failure mode this prevents:* the recurring pattern where a future agent reads `def derive_fused_partition_v2(signatures, face_classes) -> Partition` and has to grep tests for half an hour to figure out whether `face_classes` is a list-aligned-with-signatures or a dict-keyed-by-shape-id; one concrete example (`face_classes=["arc_class_A", "arc_class_A", "arc_class_B"]` aligned with `signatures[0..2]` from `tests/fixtures/petal-6-full/pattern.gt.json`) collapses that lookup to seconds. **Application surfaces:** qiyas detectors and signature compute, sacred-patterns geometric primitives, bikar DSL evaluator and gt-emitter. Where a function takes a fixture-shaped input, the example should be the fixture path; where it computes a known value, show the actual computed value.

15. **Model variant data with types and polymorphism — bag-typed dicts are forbidden, including `dict[str, object]` / `Record<string, unknown>`.** When a payload has a known variant shape (a discriminator field like `type`, `arrangement_type`, `source`; or a domain entity with subtype-specific fields), model it as a discriminated union, a class hierarchy, or per-variant TypedDict / Pydantic submodels — never as a single bag-typed dict that consumers narrow at every read site. **The fig-leaf rule:** `dict[str, Any]` is obviously wrong. `dict[str, object]` (Python) and `Record<string, unknown>` (TS) are the *same anti-pattern in disguise* — they pass the strict-mode lint by forcing use-site narrowing, but from the consumer's perspective the data is still untyped, and helper functions like `_param_int(params, "sides", 0)` that exist only to launder `object` into `int` are calcification of the anti-pattern, not a fix. **Polymorphism is part of the rule:** when shapes have an `is-a` relationship (a `HankinStarShape` is-a `Shape`; a `Polygon` is-a `Shape` with all flat edges; an `ArcShape` is-a `Shape` with at least one curved edge), prefer per-subclass dispatch over `if shape.type == X: read params[Y]` branches scattered across consumers. **Stop rule before adding `dict[str, object]`/`Record<string, unknown>` to a function signature:** check whether the data already has a discriminator field elsewhere in the schema — if it does, you're declining to use it; that's the failure. Acceptable uses: `dict[str, object]` (or `unknown`) at deserialization boundaries with *immediate* narrowing into a typed model in the same function; never propagate it through the codebase. *Failure mode this prevents:* the 2026-05-16 detector cascade where I closed ~140 strict mypy errors by swapping `dict[str, Any]` → `dict[str, object]` and adding `_param_int`/`_param_float` helpers. mypy went green, but every consumer still had to narrow at every read site — the helper functions did the narrowing instead of the type system, the discriminator (`ShapeType` Literal) was right there and unused, and the next agent adding a shape type still learns nothing from the type system about what its `params` should hold. See `qiyas/docs/decisions/2026-05-16-typed-shape-params-vs-object-bag.md` for the worked example and the four-option weighing, and `qiyas/docs/plans/geometric-class-system.md` for the companion class-hierarchy plan (Shape base / Edge / FlatEdge / CurvedEdge / Polygon = all flat edges). **Companion to tenets 2 (root cause not fallback) and 10 (FP — types over loose contracts):** this tenet names the specific failure mode for the *typing layer* — silencing the gate instead of modeling the data.

16. **Never silence strict-mode type errors with `Any`/`any` or `# type: ignore`.** When mypy strict or `tsc --strict` fires, the gate is asking "what is actually in this value?" — `Any` (Python) and `any` (TS) say "I don't know," and `# type: ignore` says "stop asking." Both defeat the gate's purpose. **The rule:** derive the real type (TypedDict, Pydantic submodel, Protocol, dataclass, class hierarchy), refactor the code that resists typing, or model the variant — but never reach for `Any`/`any`/`# type: ignore` to make the lint green. **Includes the disguised forms:** `Any` in a signature, `dict[str, Any]`/`list[Any]`, `cast(X, untyped_value)` to launder one error, `# type: ignore[arg-type]` on a single line. Each of these is a *silencer*, not a fix. **Acceptable uses:** `Any` at *parse boundaries* (JSON deserialization, plugin interfaces, untyped third-party returns) with *immediate* narrowing into a typed model in the same function; `# type: ignore` only with a code+reason comment when the type system genuinely can't express the truth (e.g., a runtime-only invariant). **Stop rule:** if your only path to making mypy/tsc green is `Any`/`any`/`# type: ignore` for data that has a known discriminator or a derivable type, stop and model the variant — that's the failure tenet 15 names from the typing-layer angle. *Failure mode this prevents:* the 2026-05-16 triangle.py edit where I closed a `dict` `[type-arg]` error by annotating `dict[str, Any]` — the gate was asking "what's actually in this dict?" and I answered "I don't know." If a function genuinely resists clean typing, that's a *signal the function does too much* — split it before adding a silencer. Cross-repo: same rule in TS — `unknown` is acceptable at parse boundaries with immediate narrowing; `any` is never acceptable.

17. **Prove the primitive before composing it — simple cases first, tiers gate higher tiers.** Before any detector / construction / spec change is measured on a composite pattern, validate it against the *simplest possible witness* — a single primitive shape, alone, with no composition, no rotation, no nesting. The failure mode this names is *composite-only validation*: you ship a detector that "works on medallion-10" but never proved it works on a single petal, so when a composite breaks you can't tell whether the bug is in detection, construction, or composition. **Tier ladder** (qiyas/calibration/i1-corpus/TIERS.md is the source of truth, but the structure is): Tier 0 = single primitives (single-petal, single-square, single-kite, single-triangle, single-pentagon, single-hexagon, single-rhombus); Tier 1 = simple compositions (two-primitive joins, single rotational pair); Tier 2 = parametric families (petal-N-ring, sunflower-N); Tier 3 = full medallions (medallion-10, hexagram-lens-full). **Gate rule:** no work on Tier N+1 entries until Tier N is at 100% detection. Any Tier 0 miss is a foundational defect and blocks all higher-tier work. **Stop rule before validating a detector/spec change on a composite:** confirm the same change has been validated on every Tier 0 primitive it touches. If not, the composite result is *ambiguous* — you don't know whether a failure is the change's fault, the construction's fault, or the composition's fault. **Companion to tenets 7 (don't tune to fit) and 8 (solve the general problem):** those say "don't overfit one case"; this says "prove the foundation before building on it." Tenet 8 names the *witness* (use 2+ members of the class); this tenet names the *ordering* (witness the atomic case before the composite). *Failure mode this prevents:* the 2026-05-04 to 2026-05-16 I1 cascade pattern where 9 corpus entries are all composite/parametric (hexagram-lens, petal-N-ring, sunflower-N) and zero are single primitives — every composite failure was diagnosed as "detector bug" or "construction bug" or "cascade bug" without a clean way to isolate which, because no Tier 0 baseline existed. The user-raised framing on 2026-05-16: *"we should prove the simple use cases then expand as a tenet... we should have done all the simple shapes — single petal, single square, kite, etc., made sure the detection properly worked and then expanded into more patterns."*

18. **Codify every debug witness as a test before the fix ships — the smoke that found the bug stays as the regression that prevents its return.** When debugging surfaces a concrete witness — a hand-run command, a one-off script, a `console.log` sweep, a smoke that revealed N=4 fills differently than N=6 — that witness is the most expensive artifact of the session. Throwing it away after the fix lands means the next refactor rediscovers the same bug with a fresh investigation. **The rule:** before any debug-driven fix ships (commit, PR, merge), the witness that justified it MUST exist as a unit/integration test in the appropriate suite. The test asserts the invariant the witness probed — not the symptom narrative, the actual measured property (face count, fill count, classification verdict, signature value, whatever the smoke produced). **Stop rule before committing a debug-driven fix:** name the test that encodes the witness. If you can only name the shell command or the printf you ran, you haven't finished — write the test, run it (red → green confirms the witness is real), then commit fix + test together. **Acceptable deferral:** if the test requires non-trivial scaffolding (a new fixture, a corpus regen, a cross-repo data dependency), file the test as a tracked follow-on task *blocked by* the fix's commit AND mention the task ID in the commit message — the witness still lives, just in the task graph instead of the test suite, with an owner. **Companion to tenets 4 (verify before claiming done), 7 (don't tune to fit), and 17 (prove the primitive first):** tenet 4 says verify your assertion; tenet 7 says don't make a symptom-fix that only works on one input; tenet 17 says start from the atomic witness; this tenet says *the witness you used must outlive the session*. Together they're a four-step loop: write the simplest witness (17) → verify the fix against it (4) → confirm it generalizes (7) → freeze the witness as a test (18). *Failure mode this prevents:* the 2026-05-18 bikar#333 Option E session — the smoke `for N in 4 6 8 10; do render petal-N-ring; grep -c 'fill="#1B6CA8"' out.svg; done` revealed exactly the source-tag-path interaction the decision doc didn't anticipate (2N stamps when expected N), but the smoke lived only in shell history. Without this tenet, the next refactor of `assignFaceClasses` would rediscover the same bug from scratch. With it, the petal-N-ring-class-count test is filed *as part of* the Option E shipment (task #413), not after, and the regression can never silently return. *Anti-pattern this names:* "I ran the smoke, the fix works, ship it" — that workflow is exactly the "verify but don't codify" gap; the cost of authoring the test is paid once, the cost of not authoring it is paid by every future agent who reads the code.

19. **Bias for action — make the most practical decision, log it, raise only when going in circles.** When work surfaces a choice between paths and the practical answer is reachable from current evidence + the tenets, **pick it and act**. Don't surface the choice to the owner for confirmation; don't ask permission to apply an existing decision doc's recommendation; don't pause work to enumerate options the owner already endorsed implicitly through prior direction. The **only required artifact** is logging the decision: if it's a load-bearing pick, write a one-paragraph decision note (commit message, issue addendum, or new decision doc per the decision-doc skill if it crosses a boundary that matters); if it's a routine pick, mention it in the commit message. **Stop rule — raise to owner only when:** (a) you've made the same call twice and reversed it (going in circles), or (b) the practical answer requires a destructive action the owner hasn't pre-authorized (rm -rf, force-push, public-facing send, accepting a decision-doc on behalf of another decision's owner), or (c) two equally-weighted paths exist AND the evidence to choose can only come from the owner's intent (preference, scope, deadline). **Companion to tenet 12 (commit autonomously on tested chunks) and tenet 13 (correct over quick):** tenet 12 says "ship the work without permission"; this tenet says "make the call without permission." Together they're the autonomy contract — the owner directs the loop and reviews diffs, not picks every routing decision. *Failure mode this prevents:* the 2026-05-18 session where I walked through 6 decision docs in detail then asked "should I file this as task A or task B; should I accept Option D on your behalf or leave it PENDING" — three confirmation rounds where the practical answer ("accept the existing decision doc that already recommends what the owner asked for, file the task, start the work") was reachable from the conversation. The cost of each unnecessary question is one owner turn-around and one stalled loop iteration. *Anti-pattern this names:* presenting menus when the answer is on the menu; phrasing "do you want me to X or Y" when X is the obviously practical move; treating every cross-repo task filing as a fresh permission ask instead of the routine work it is.

20. **Fix the simplest broken thing first — work up the complexity ladder, never down it.** When the backlog has multiple broken things across tiers (single primitives, simple compositions, parametric families, full medallions), sort candidates by tier and pick the lowest-tier broken thing as the next fix — regardless of perceived impact, regardless of which one surfaced most recently, regardless of which one a downstream consumer is asking about. A Tier 0 single-primitive miss is foundational: every Tier ≥1 construction that uses that primitive is silently corrupted, every detector measurement against it is multi-layer ambiguous, every "fix" you ship at Tier 3 is built on quicksand if the Tier 0 it depends on is still broken. **Stop rule before picking the next fix from any backlog (loop pickup, decision-doc shipment, issue queue):** enumerate candidates with their tier (use `qiyas/calibration/i1-corpus/TIERS.md` as the canonical ladder), pick the lowest tier. If you find yourself reaching for a higher-tier fix because "the primitive is just a special case" or "the rhombus regression is upstream's problem," stop — that's the failure mode this tenet names. **Companion to tenet 17 (prove the primitive before composing):** tenet 17 gates *validation* (don't measure a detector against composites before primitives are green); this tenet gates *remediation* (don't fix composites before primitives are green). Together they're a two-sided constraint: foundations green before measuring up, foundations green before fixing up. *Failure mode this prevents:* the 2026-05-18 session where I shipped §B6 Option B for star7 (Tier 3, full medallion) without noticing single-rhombus (Tier 0) had silently regressed — bikar#407's edge-identity probe dropped the rhombus face entirely (3 shapes → 2 circles, no rhombus). The user had to ask twice ("did we get the simple shapes right?", "did we add them to our reports?") before I checked. Simplest-first ordering would have queued the rhombus regression ahead of star7's medallion-level Option B prototype, caught the Tier 0 foundation breaking *before* declaring a Tier 3 victory, and avoided the false-progress signal. *Anti-pattern this names:* "let me finish the cool composite work first, then loop back to the primitives" — the primitives never get the loop-back because composites always have more visible deliverables; sequencing them last guarantees they stay broken.

21. **A narrow corpus sweep is not the acceptance bar for a constant tweak — run the full suite, every time.** When a fix takes the shape "change a numeric constant" (tolerance widening, threshold loosening, epsilon multiplier, fold-symmetry slack, detector confidence floor), do NOT treat a hand-picked corpus sweep — even a comprehensive one — as the "no regression" evidence. The minimum acceptance bar is full `vitest run` / `pytest` / equivalent. Corpus templates are a *sample* of the input space; synthetic test fixtures probe edge cases the corpus deliberately avoids. A constant change that passes the corpus but fails the synthetic suite is the canonical "looked safe, silently broke a different invariant" failure. **Stop rule:** before declaring a constant tweak shippable, name the full test command you ran and confirm it was the full suite, not a curated subset. If you only ran a corpus sweep, you haven't cleared the bar — run the suite. If the suite reveals a regression, the constant-tweak approach is *falsified*; do NOT pivot to "per-call-site tolerance" or "scoped tolerance" (that's the second tweak tenet 7 names as the stop signal). Author the decision doc for the architectural fix instead. **Companion to tenets 4 (verify before claiming done), 7 (stop after 2nd tweak), and 17 (prove the primitive first):** tenet 4 says verify your own claims; tenet 7 says stop tweaking after two; tenet 17 says the *primitive* test is non-negotiable; this tenet names the specific class of evidence — full-suite, not corpus-subset — that constant tweaks require. *Failure mode this prevents:* the 2026-05-19 bikar#424 Phase 1c session where I widened `PointIndex(EPSILON*100)` → `PointIndex(EPSILON*1000)` to fix N∈{8,12,16} petal-N-ring face extraction, ran a corpus sweep across hexagram-lens, medallion10-iter14, petal-6-full, single-petal, single-rhombus, saw zero regression, and concluded "Option D ships." The wider full vitest sweep — which I'd skipped before declaring success — surfaced `packages/core/tests/kernel/petal-debug.test.ts > layer 8 inner-triangle face` regressing: the wider tolerance dissolved 3 layer:8 faces in an N=6 pattern by over-merging endpoint vertices with adjacent intersection points. The corpus sweep missed it because petal-debug's layer:8 pattern is a synthetic fixture, not a corpus template. The owner caught the gap directly: *"changing tolerance doesn't feel like we are fundamentally solving problem... don't we have a tenant against going in circles in this regards"* — invoking tenet 7. The right move was the full suite first, then revert and pivot to Option C (preserve curve identity through the merge). *Anti-pattern this names:* "I swept a few representative patterns and nothing broke" — representative ≠ exhaustive; synthetic fixtures exist precisely because corpus templates don't probe the failure modes those fixtures encode.

22. **Consumer audits must cover construction-time contracts, not just graph-walk contracts — audit upstream producers, not only downstream consumers.** When auditing whether a kernel-level refactor (vertex-merge policy, edge-key strategy, canonicalization rule, signature schema migration) is safe, the standard step — read every downstream consumer of the kernel's output and verify the refactor doesn't break their contracts — is **necessary but not sufficient**. The audit must also cover the **upstream construction code** (DSL evaluators, primitive emitters, gt-emitter, signature compute callers, orchestrator wrappers) that produces the kernel's *inputs* with implicit invariants baked in. **The failure mode this names:** an audit that passes on the output side declares "safe to refactor"; the input side relies on an implicit shared-vertex / shared-identity / shared-tag contract the refactor breaks; every healthy baseline regresses on the first composite run. **Stop rule before declaring an audit complete:** the audit verdict must explicitly enumerate (a) the downstream consumers it covered, AND (b) the upstream producers it covered, AND (c) the implicit invariants those producers rely on the kernel preserving. The sentence "this audit covers EdgeGraph consumers but does not cover EdgeGraph producers" is the load-bearing addendum that prevents the false verdict — write it explicitly or expand the audit. Before shipping the refactor, run a hand-picked construction-time fixture (the simplest one that exercises the implicit invariant) — not a corpus sweep, not the full suite, the *atomic primitive* that depends on the invariant; if that fails, the refactor is falsified at the construction-time layer regardless of what the consumer audit said. **Companion to tenets 4 (verify before claiming done), 7 (stop after 2nd tweak), 17 (prove the primitive first), and 21 (full-suite acceptance):** tenet 4 says verify your own claims; this tenet says verify the *audit's* claims by exercising the construction side; tenet 7 names when to stop iterating local probes; tenet 17 names the witness ordering (atomic before composite); tenet 21 names the post-fix acceptance bar. Together they form a coverage matrix: input invariants × output contracts × atomic × composite × suite. *Failure mode this prevents:* the 2026-05-19 bikar Option C cascade. The slice-1 audit (`bikar/.claude/plans/option-c-consumer-audit.md`) read all 5 EdgeGraph consumers (planar-graph-builder, face-extractor, polygon-union, evaluator, strapwork) and produced a verdict of "safe to refactor — preserve curve identity through vertex merge." Slice-2 implementation revealed the verdict was wrong: every petal-N-ring construction in the corpus relies on two arcs from *different circles* having *intentionally shared endpoint coordinates* — that's the entire mechanism by which a "lens face" closes between two circles. The DSL emits `connect arc P1 -> P2 on circleA` plus `connect arc P1 -> P2 on circleB` where P1 and P2 are arc-arc intersection points; the implicit contract is "these endpoints must canonicalize to the same vertex index in the planar graph." Option C un-merged them, the lens faces dissolved, every healthy baseline regressed. Three Option-C variants tried in one session, all falsified. The audit missed this because it only checked the *output* side — the construction-time contract was implicit in the DSL evaluator and gt-emitter, not in any consumer of the EdgeGraph. **Cross-repo:** the same trap exists for qiyas detector refactors (does the detector audit cover the signature-compute callers that feed it?) and for sacred-patterns iteration-loop refactors (does the orchestrator audit cover the validate-svg preflight and qiyas-diff wrapper?). Audit both sides.

23. **DSL-as-source-of-truth — when the authoring layer knows a fact, propagate it as authoritative; don't re-derive downstream.** When a fact about the geometry is *known at DSL-authoring time* (face class, symmetry fold, construction-source identity, the side count of a polygon the user explicitly stamped), that fact MUST propagate through the SVG `data-*` attributes (or equivalent intermediate-representation channel) and be consumed downstream as authoritative — not re-derived from jittered geometry by a detector that has to second-guess what the DSL already declared. Re-derivation is the *fallback* for inputs without a producer (photos, traced raster) — never the default for inputs that have one. **Stop rule before adding a tolerance, classifier branch, or re-derivation helper to a downstream consumer:** ask "did the producer already know this?" If yes, the fix is in the channel (extend the data-* contract, plumb the field through), not in the consumer's heuristic. If you're tempted to bump a tolerance because the consumer can't tell two cases apart, that's a missing channel, not a missing tolerance — every tolerance bump in service of a fact the producer knew is a Tenet 7 / Tenet 21 violation in disguise. **The DSL Metadata Contract** (`sacred-patterns/docs/dsl-metadata-contract.md`) is the versioned, cross-repo enumeration of every fact the producer carries and every consumer field that reads it — entries enter the contract only when a named downstream call site would otherwise re-derive them; the contract doc PR requires citing the consumer to prevent over-tagging. **Photo-cascade carve-out:** for inputs without a producer (I2 photo cascade — camera + tracer + detector), the qiyas detector becomes the producer-of-record; its tags carry detector-confidence (not authorship-confidence) and the matcher cost-weights accordingly per the DETR pattern. The tenet is "when the producer knows, trust it" — not "everything is tagged." **Companion to tenets 2 (root cause not fallback), 7 (don't tune to fit), 11 (one tool path per question), and 15 (typed variants):** tenet 2 says don't add fallback paths; tenet 7 says don't tune constants for one input; tenet 11 says don't fragment tooling; tenet 15 says model variants in types; this tenet says the same failure shows up at the *cross-layer-channel* level — when a downstream consumer re-derives instead of reading from an upstream channel, the noise floor it absorbs is the cost of the missing channel, paid forever. *Failure mode this prevents:* the 2026-05-20 qiyas#490 anti-symmetry breach (`qiyas/docs/decisions/2026-05-20-qiyas-anti-symmetry-floor-breach.md`) — the matcher cross-paired Star-8 squares against Hexagram triangles because bikar's DSL knew `face_class=square` vs `face_class=triangle` at authoring time and the channel didn't carry it, so qiyas had to re-derive from contour vertex counts that jittered enough to be interchangeable. The first reflex was a tolerance-style cost penalty (Option A) — that's the in-disguise tenet-7 violation. The structural fix is the channel: emit `data-face-class` from bikar, read it in qiyas, drop the re-derivation. Sibling instances: qiyas#400 (vocabulary lumping — squares to `unknown` because the polygon classifier re-derived sides from jittered vertices), the symmetry pillar mis-scoring (qiyas's `_check_dominant_fold` re-derives fold from rotational orders instead of reading `data-symmetry-fold`). All three are the same defect class.

When the right move is to violate a tenet, say so explicitly and explain why — don't violate quietly.

## Architecture

### Source Layout

All TypeScript source lives in `src/ts/`. Webpack entry point is `src/ts/index.ts`, which re-exports drawing functions that compose the geometry primitives.

### Geometry Primitive Layer (`src/ts/`)

- **`points.ts`** - `Point` class (2D coordinates, distance, quadrant detection)
- **`lines.ts`** - `Line` class (slope, extension, scaling); `Lines` factory
- **`circles.ts`** - `Circle` class (circumference points, surrounding circles, recursive flower patterns); carries metadata (level, fill, stroke)
- **`polygons.ts`** - Base `Polygon` class with subclasses: `Triangle`, `Square`, `Pentagon`, `Hexagon`, `Heptagon`, `Octagon`, `Nonagon`, `Decagon`; chord/sagitta calculations
- **`star.ts`** - `Star` class, `FivePointStar`, `ElongatedFivePointStar`

### Visualization Layer

- **`canvas.ts`** - D3.js SVG rendering functions (`appendCircle`, `appendLine`, `appendPolygon`, `appendText`); gradient definitions; type aliases `d3SVG` and `d3CIRCLE`
- **`index.ts`** - High-level drawing functions that compose primitives into patterns (e.g., `drawCirclesRecursively`, `drawChainedStars`, `drawStarGrid`); SVG DOM creation and color utilities

### Supporting

- **`helpers.ts`** - Functional utilities (`all`, `any`, `isEven`, `isOdd`, `_map_even_odd`, `applyTransformationPipeline`)
- **`types.ts`** - `IO` (void alias), `Optional<T>`

### Build Pipeline

TypeScript (`src/ts/`) → Webpack + ts-loader → UMD bundle (`site/bundle.js`) + HTML from template (`templates/index.tpl`) → static site served from `/site` directory

D3 and Lodash are webpack **externals** (loaded via `<script>` tags, not bundled).

## Key Conventions

- Geometric transforms are **immutable** — methods like `rotate()`, `adjacent()`, `scaleLine()` return new instances
- Objects carry rendering metadata (level, stroke, fill) through transformations
- Circles serve as reference frames for positioning surrounding shapes
- Heavy use of Lodash (`_`) for functional operations
- TypeScript strict mode is enabled
- ESLint uses `@typescript-eslint` parser with recommended presets (flat config, eslint.config.js)
- **Geometric construction from central origin** — All proportions are derived from trigonometric relationships relative to a central circle, not arbitrary scaling factors. Radii come from star polygon edge intersections (`cos(pi*k/n)` formulas), positions from polar coordinates at `360/n` intervals, and distances from additive geometric terms (tip radius + band width + satellite radius). Never use unexplained `R * 0.52`-style constants.
- **Relative-to-origin scaling** — Every coordinate, distance, and size in a pattern MUST be defined as a ratio or function of the originating circle's midpoint `(cx, cy)` and radius `R`. When `R` changes, the entire diagram scales proportionally — no element has an absolute position or size. This mirrors the landing page implementation where `surroundingCircles(6, 1)` places circles at `R` from center with radius `R`, and nonagons are sized at `0.75 * R`. Satellite distances, band widths, tile vertices — all expressions of `R`, never standalone constants.

## Workflow Conventions

- **Task tracking** — When the user issues a command (implement, fix, refactor, create, etc.), track progress with tasks naturally. When the user asks a question or wants to discuss something, engage in conversation first — don't create tasks until a course of action is agreed upon.
- **Feature documentation** — User-facing features for interactive tools (toolbar buttons, keyboard shortcuts, selection modes, etc.) are documented in `docs/features.md`. Update this file when adding or changing UI features.
- **Issue documentation** — When work surfaces a non-trivial issue with more than one plausible fix (parity divergence, dependency gap, design assumption challenged by evidence), file it as a markdown doc under `docs/issues/` (or, when the issue belongs to a sibling repo like qiyas, under that repo's `docs/issues/`). Required structure: **Status** (OPEN / RESOLVED YYYY-MM-DD / WONTFIX), **Discovered** (when, by whom, what surfaced it), **Symptom** (concrete observation with reproducer), **Root cause** (established by investigation, not speculation), **Options considered** (every option with pros/cons/cost — including the one not picked), **Decision** (which option, why, who decided), **Follow-ups** (task IDs, backlog entries). Filename: `YYYY-MM-DD-short-slug.md`. The "Options considered" section is load-bearing — the point of filing is so future readers can see what was weighed, not just what was picked. Don't delete resolved issues; they're the archive. Skip this for trivial bugs (just fix), scope deferrals (track in backlog docs), or open design questions before any work has started (use plans).
- **Cross-repo task linkage** — When a sacred-patterns task blocks or is blocked by work in qiyas or bikar, mention it in the task description as `qiyas#NN` or `bikar#NN` (grep-able). Active cross-repo dependencies live in `docs/cross-repo-dependencies.md` (mirror of the same file in qiyas and bikar). Prune resolved rows once both sides ship.
- **Task hygiene — archive then delete completed tasks** — The task list is for **in-flight** work. Completed tasks consume context on every load and obscure what's actually unblocked. **Policy:** when ≥20 completed tasks have accumulated OR a cascade closes, batch-archive them to the cross-repo `CHANGELOG.md` (root of sacred-patterns; mirrors in qiyas and bikar) and delete them via `TaskUpdate status: "deleted"`. **Audit before deleting:** confirm no open task lists a completed task in its `blockedBy` — if so, the open task is stale (update its description to reference the CHANGELOG entry, then drop the blocker). The CHANGELOG is a **durable index into decision docs, plans, and commits per Tenet 6** — entries name the outcome and cross-reference the doc/commit that holds the *why*; never inline the rationale. Format: `#NN [repo-prefix] subject — one-line outcome`; group by cascade, not chronology. Skip archival for trivially-scoped tasks that left no decision-doc footprint (just delete) and for tasks whose completion is captured by a single commit message (commit hash is enough).

## HARD GATES — Iteration Rules (learn-new-pattern skill)

These rules are ENFORCED as hard gates during iteration work. They survive conversation summarization because they live here in CLAUDE.md.

### G1. Iteration Immutability

**Once `iterations/{N}/output.html` renders successfully in the browser (no JavaScript console errors), that folder is FROZEN.** Any further code change — no matter how small — MUST go in `iterations/{N+1}/output.html`.

**Pre-edit checklist (MUST run before editing ANY output.html):**
1. Has this file already rendered successfully? Check: did I take a screenshot or see console output from this file?
2. If YES → STOP. Create `iterations/{N+1}/`, copy output.html there, edit the COPY.
3. If NO (first render attempt, or fixing JS errors on a broken render) → proceed with edits.

**The ONLY exception:** Fixing JavaScript errors that prevent the file from rendering at all (blank screen, console errors). Once it renders visual output, it is frozen.

### G2. Architecture Before Pixels

If the structural audit score is < 100% (ANY element MISSING or MALFORMED), the next iteration MUST target a structural improvement. Pixel-focused iterations (color tweaks, opacity adjustments, proportion tuning) are ONLY permitted when structural score = 100%.

### G3. Mandatory Deliverables Per Iteration

Every iteration MUST produce these files BEFORE starting the next iteration:
- `evaluation.md` — Structured comparison against reference
- `retrospective.md` — What worked, what didn't, why
- `guidance.md` — What to change in the next iteration

**Never skip straight from render to code changes.** The evaluate→reflect→plan cycle is what prevents repeated mistakes.

### G4. Reference-Grounded Verification

The structural audit MUST be grounded in the reference image, not self-defined labels. For every element:
1. Look at the specific zone in the REFERENCE first
2. Then look at the same zone in the OUTPUT
3. If they don't match, the element is MALFORMED — even if the output has "something" in that zone

### G5. Skill-Prefixed Task Naming

When creating tasks for iteration work, always prefix the task subject with `[learn-new-pattern]` to associate it with the skill and survive conversation summarization.

## Convergence Philosophy — Shape Before Pixels

Pattern convergence is measured by **structural and geometric fidelity**, not pixel-similarity metrics. After 90+ iterations on the first pattern, we learned that VTracer/JPG similarity percentages plateau early while significant architectural differences remain. Metrics can be gamed by color/coverage without matching actual shapes.

### Convergence Hierarchy (evaluate in this order)

1. **Topology** — Correct number and connectivity of structural elements (star, kites, petals, satellites, interstitial tiles, strapwork, interlacing). Tracked via the C1 structural audit (7/7 for this pattern).
2. **Geometry** — Individual tile shapes match the reference (correct vertex count, angles, proportions). Compare per-element: overlay one tile type at a time against the reference.
3. **Proportions** — Relative sizes, distances, and ratios match (rosette-to-satellite ratio, band width, star tip depth). Verify with geometric analysis tools.
4. **Coloring** — Last priority. Once shapes are correct, match reference colors.

### Architectural Convergence Audits (A1-A6)

Every iteration must report these structural metrics (defined in iteration-guide.md):

- **A1. Structural Census** — SVG element counts + arrangement metrics from console logs. Track deltas.
- **A2. N-fold Symmetry** — Verify all sectors are identical (EXACT/APPROXIMATE/BROKEN).
- **A3. Tile Shape Distribution** — Classify faces by vertex count, compare to reference expectations.
- **A4. Coverage Completeness** — Is the medallion fully tiled? (FULL/PARTIAL/SPARSE).
- **A5. Band Network Integrity** — 10 continuous bands, correct crossings, consistent width (COMPLETE/GAPS/BROKEN).
- **A6. Baseline Shape Validation** — Do shapes match user-verified expectations? (PASS/PARTIAL/MISSING per shape). Requires `--baseline input/baseline.json` (produced by the interpret-pattern skill). When A6 reports MISSING or PARTIAL, the G2 gate forces the next iteration to target that shape.

These audits measure shape convergence directly, unlike pixel metrics which conflate color, coverage, and structure.

### Pixel Metrics as Regression Detection

- **`qiyas pixel-diff`** scores are useful for detecting regressions (did this change make things worse?) but NOT for guiding structural work
- A change that adds correct structure may temporarily decrease the similarity score
- Never optimize for the metric — optimize for visual/structural match against the reference
- The primary progress indicators are: `validation.json` → `overall.warnings[0].counterfactual_score_delta` (predicted next-step score lift, ranked) and `overall.structural_score` (A6 PASS count). Pixel similarity is diagnostic-only.

### Iterations Are Driven by `overall.warnings[0]`

Every iteration's priority comes from `validation.json`'s `overall.warnings[0]` — qiyas has already ranked the warnings by predicted score lift (`counterfactual_score_delta`). The iteration agent translates `warnings[0]` into a code change; it does not re-derive priorities from screenshots. Each warning carries `context.counterfactual_rationale` ("if X were Y, score would improve by Z") — that rationale IS the proposed fix. See `iteration-guide.md` sections C0 (Read warnings) and E2 (Mechanical edit derivation).

## Self-Healing Principles

These patterns prevent wasted iterations and catch problems early:

1. **Verify before evaluating** — After rendering output in the browser, always check the console for JavaScript errors (`mcp__chrome-devtools__list_console_messages` with `types: ["error"]`). Never evaluate blank or broken output — fix the render first.

2. **Diagnose before retrying** — When output is wrong, understand WHY before rewriting. Read the previous retrospective. Identify the root cause (wrong approach, missing variable, wrong geometry) rather than making the same structural mistake with different code.

3. **Simplify on failure** — If a complex algorithmic approach produces wrong results, step back to a simpler, more direct approach. Over-engineered solutions are harder to debug and more likely to have subtle bugs.

4. **Compare per-element** — Don't evaluate everything at once. Compare one element at a time (central star, satellites, interstitial tiles, strapwork) against the reference. Fix the highest-impact element first.

5. **Track changes explicitly** — Every iteration must clearly state what changed and why (`guidance.md`). This prevents regression where fixing one thing breaks another.

6. **Consult learnings first** — Before starting work, read existing learnings and common mistakes. Many errors are recurring patterns that can be avoided.

7. **Validate SVG artifacts** — When extracting SVG from D3.js renders (via `document.querySelector('svg').outerHTML`), the SVG requires post-processing. After saving, always run `./tools/validate-svg.sh output.svg` which checks: starts with `<svg`, ends with `</svg>`, has `xmlns`, valid XML, contains visual elements, no wrapper artifacts. Key fixes:
   - **Add `xmlns`**: `sed -i '' 's/<svg /<svg xmlns="http:\/\/www.w3.org\/2000\/svg" /' output.svg`
   - **Strip wrapper text**: If tool output has markdown/JSON wrappers, extract the raw SVG string first.
   - **Curved lines are circles**: Any curve in a geometric pattern is an arc of a circle. Identify the center and radius, express as `f(R, n, k)`, and draw with exact circular arcs — never Bezier approximations.
