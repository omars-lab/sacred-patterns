# Iteration Guide

Core workflow for iteratively replicating a reference pattern. Each iteration produces a self-contained D3.js HTML file, evaluates it against the reference, and applies corrections.

## Iteration Folder Structure

Each iteration lives in `iterations/{nn}/` (zero-padded: 01, 02, etc.):

```
iterations/01/
  pattern.json          # Pattern definition for this iteration
  output.html           # Self-contained D3.js v7 HTML (supports ?mode=blueprint and ?mode=final)
  output.svg            # SVG extracted from the D3 render (for programmatic comparison)
  output-tiles-only.svg # SVG with strapwork overlay disabled (for tile geometry metric)
  screenshot.png        # Browser screenshot (final mode — fills + strapwork)
  blueprint.png         # Browser screenshot (blueprint mode — edges only, no fills)
  reference.jpg         # Copy of reference image (so each iteration is self-contained for comparison)
  reference.svg         # VTracer-traced reference SVG (copied from input/reference-traced.svg)
  validation/
    svg-audit/
      svg-audit.json    # Machine-readable A1/A2/A4/A5/A6 audit data from qiyas svg-audit
      encoding.json     # qiyas encoding produced from output.svg (input to svg-audit)
  diff/                 # Diff artifacts (generated by qiyas pixel-diff via Docker)
    diff-visual.png     # Diff overlay (from vs-vtracer, for backward compat)
    diff-heatmap.png    # Pixel difference heatmap
    diff-stats.md       # Numeric similarity metrics
    diff-a-norm.png     # Normalized output render
    diff-b-norm.png     # Normalized reference render
    vs-vtracer/         # Diff against VTracer SVG (spatial structure metric)
    vs-jpg/             # Diff against original reference JPG (visual fidelity metric)
    vs-vtracer-tiles-only/  # Diff of tiles-only SVG against VTracer (tile geometry metric)
    vs-jpg-tiles-only/      # Diff of tiles-only SVG against JPG (tile visual fidelity)
  evaluation.md         # Structured comparison (uses BOTH screenshots + SVGs + diff)
  retrospective.md      # Why did analysis lead to this output?
  discoveries.md        # Updated snapshot of confirmed facts, assumptions, open questions
  guidance.md           # What to change next
  library-changes.md    # Any TS library changes (if applicable)
```

### Comparison Artifacts Rule

Every iteration directory MUST contain both the output AND the reference for side-by-side comparison:

1. **`output.svg`** — Extracted from the D3 render. After taking screenshots, run JS in the page to extract the SVG DOM:
   ```javascript
   document.querySelector('svg').outerHTML
   ```
   Save the result to `iterations/{nn}/output.svg`.

2. **`reference.png`** — Copy of `input/reference.{ext}` into the iteration folder. This makes each iteration self-contained — you can compare output vs reference without navigating to the session root.

3. **`screenshot.png`** + **`blueprint.png`** — As before, browser screenshots of final and blueprint modes.

**Why:** Without these artifacts, evaluation depends on having the reference open in a separate window and mentally comparing. With them, any iteration folder contains everything needed for comparison — useful for context resets, retrospective review, and programmatic SVG diffing.

### Stagnation Detection

Before starting any new iteration, check whether progress has stalled. Stagnation wastes iterations and compounds bad approaches.

**Structural stagnation rule (takes priority):** If the structural audit (C1) has the same MISSING/MALFORMED elements for 3+ consecutive iterations, STOP. The construction approach is fundamentally wrong — no parametric tuning will add missing elements. Perform a Brutal Honesty Checkpoint (Section I) immediately.

**3-iteration lookback rule:** Compare the similarity scores of the last 3 iterations. If the delta across those 3 iterations is less than 1.0%, STOP and perform the stagnation analysis below before proceeding. Do not write `guidance.md` or start a new iteration until this analysis is complete.

**Stagnation analysis checklist** (work through all five):

1. **Are the pixel numbers actually changing?** Check shared content %, only-in-output %, and only-in-reference % across the last 3 `diff-stats.md` files. If they are identical or near-identical, the output geometry is not changing — parametric tweaks are not working and the approach needs to change.
2. **Is the diff structural or localized?** Open the last 3 `diff-heatmap.png` files. Uniformly red = global structural mismatch (wrong topology, wrong construction). Localized red spots = specific element errors (fixable with targeted changes). Structural problems cannot be solved by parameter tuning.
3. **Has the approach been exhausted?** Read the last 3 `guidance.md` files. If they are variations of the same approach (same algorithm, same construction, just different parameters), that approach is exhausted — pivot to something fundamentally different.
4. **Is research overdue?** Check when external research (web search, paper review, reference implementation study) was last done. If more than 5 iterations ago, do research before the next iteration.
5. **What is the single most obvious visual difference?** Open `diff-a.png` (output render) and `diff-b.png` (reference render) side by side. Identify the one most visually striking difference. That is the priority for the next iteration, regardless of what metrics suggest.

**Anti-stagnation actions** (in priority order):

1. Search the web for how other implementations solve the specific problem you are stuck on.
2. Study reference implementations (Kaplan's Taprats, CodingTrain/StarPatterns, etc.) for the pattern type.
3. Simplify: revert to the last iteration that showed measurable improvement and try a fundamentally different approach from there.
4. Visual-first diagnosis: look at the actual rendered images side by side, not just similarity scores.

**Guidance confidence check:** Before writing `guidance.md`, verify that the proposed approach has NOT been tried (or a close variant) in the last 5 iterations. If it has, the guidance must explicitly state why this attempt will produce a different result — otherwise it will not.

### Discovery Tracking (`discoveries.md`)

Every iteration must include an updated `discoveries.md` — a running document that captures our growing understanding of the pattern. It has five sections:

1. **Confirmed Facts** — Things we KNOW (proven by analysis, output matching, or geometric derivation). Example: "The pattern has 10-fold symmetry (D10)" or "Diagonal bands follow {10/4} star polygon topology."

2. **Working Assumptions** — Things we BELIEVE but haven't fully confirmed. Each should note what would confirm or disprove it. Example: "The interstitial tiles are kite-shaped (would be confirmed by computing line-line intersection angles)."

3. **Disproven Assumptions** — Things we THOUGHT were true but discovered were wrong. Include the iteration that disproved them. Example: "Assumed radial-network spoke-and-ring topology → disproven in iter 31b (reference uses {10/4} chord topology)."

4. **Open Questions** — Things we DON'T KNOW yet. Example: "Are the interstitial shapes explicitly drawn tiles or negative space between overlapping layers?"

5. **Negative Space Observations** — Shapes that appear to be formed by VOIDS between other shapes rather than being explicitly drawn. In Islamic patterns, many apparent "tiles" are actually the background showing through gaps between overlapping star polygons, rosettes, or interlace bands. Before drawing a shape as an explicit tile, ask: "Could this shape emerge naturally if I just overlap the surrounding shapes correctly and let the background color show through?"

The session-level `discoveries.md` is the canonical version. Each iteration's copy is a snapshot. When starting a new iteration, copy the session-level file and update it with new findings.

## Geometric Pre-Analysis

**Before the first iteration**, open the analysis tools in `tools/analysis/` to establish the geometric framework. This prevents wasted iterations by revealing the pattern's structure upfront.

### Pre-Analysis Steps

1. **Star Intersection Map** (`tools/analysis/star-intersection-map.html?n=N&k=K&R=R`)
   - Map all diagonal intersections and their radii
   - Identify parallel diagonal pairs (these create no interstitial intersections)
   - Record findings in `discoveries.md` under "Confirmed Facts"

2. **Sector Decomposition** (`tools/analysis/sector-decomposition.html?n=N&k=K&R=R&bandWidth=BW`)
   - For each sector gap, determine if bands cross or are parallel
   - If parallel → face decomposition won't work in interstitial zone, use Strategy 9
   - If crossing → face decomposition can find interstitial tiles directly

3. **Tile Decomposition** (`tools/analysis/tile-decomposition.html?n=N&k=K&R=R&sector=0`)
   - Compute exact tile vertices for sector 0
   - Identify all tile shapes (kite, rhombus, petal, wing, between-band)
   - Use vertex coordinates as ground truth for iteration code

4. **Construction Overlay** (`tools/analysis/construction-overlay.html?n=N&k=K&R=R`)
   - Verify the full pattern with all layers toggled on
   - Use layer toggles to isolate and understand each structure

Record all pre-analysis findings in the session's `discoveries.md` and reference them in `analysis.md`.

See `tools/analysis/README.md` for full parameter reference and workflow details.

### Growing the Analysis Toolkit

During the iteration loop, when you discover a geometric computation that would be useful across sessions (not just this pattern), extract it into a new tool in `tools/analysis/`:

1. Create a self-contained HTML file following the same pattern as existing tools
2. Add URL query parameters for all geometric inputs
3. Verify in browser (check console for errors, take screenshot)
4. Update `tools/analysis/README.md` and `tools/analysis/showcase.html`
5. Add to SKILL.md Guide Files table
6. Deploy with `make deploy`

Don't build tools speculatively — build them when iteration work reveals a real need. If you compute something ad-hoc more than twice, it should be a tool.

## Core Tenets

These tenets govern ALL iteration work. They are ordered by priority — when in conflict, higher tenets override lower ones.

### T1. Structure before pixels

**The primary measure of progress is structural fidelity, not pixel similarity.** A 60% pixel match with correct rosette topology is closer to the goal than an 80% pixel match achieved by painting the right color blobs without any real geometry. Pixel diff is a secondary sanity check, never the primary evaluation. Every iteration must answer: "Does our output have the same architectural elements as the reference?" — not "Did the similarity percentage go up?"

### T2. Count the elements

Before evaluating colors, proportions, or pixel matches, **count the structural elements** in both the reference and the output:
- How many rosettes? (reference: 1 central + 10 satellites = 11)
- How many distinct tile shapes per rosette? (star + kites + petals = 3 layers)
- How many strapwork band crossings?
- How many total tile regions?

If the counts don't match, no amount of color tuning will close the gap. Fix the element count first.

### T3. A rosette is three concentric layers

In Islamic geometric patterns, a rosette is NOT a circle with wedge divisions. A rosette has:
1. **Inner star polygon** ({n/k} star) — the dark center
2. **Kite ring** — n kite/almond tiles radiating between star points
3. **Hex petal ring** — n elongated hexagonal petals forming the outer boundary

If the output doesn't show these three layers in both the central and satellite rosettes, the architecture is wrong regardless of what the pixel metric says.

### T4. Strapwork is structure, not decoration

The white strapwork bands are the SKELETON of the pattern — they define every tile boundary. The tiles are the colored regions BETWEEN bands. Strapwork is not an overlay applied after the fact. The correct construction order is:
1. Draw the strapwork band network (following {n/k} diagonal lines)
2. The tiles are the faces of the planar graph formed by band edges
3. Color the faces

Strapwork should only be rendered when its geometry matches the reference (see Self-Healing Pattern #8).

### T5. Structural audit gates pixel optimization

Do NOT spend iterations on color refinement, pixel optimization, or parametric tuning until the structural audit passes. The structural audit (Section C1) must show all architectural elements PRESENT before moving to color/proportion refinement. Iterating on colors while missing entire rosettes or strapwork is wasted work.

### T6. Architecture advances before pixel optimization

**If structural score < 100% (i.e. ANY element is MISSING or MALFORMED), the next iteration MUST target a structural improvement.** Pixel-focused iterations (color tweaks, opacity adjustments, proportion tuning) are ONLY permitted when structural score = 100%. This is a hard gate — no exceptions.

**Why:** It is tempting to "polish" what exists rather than tackle the harder structural work. Iterations 81-82 demonstrated this trap: two iterations spent on band opacity and color variation while interlacing remained missing. Both regressed. Structural gaps dominate the metric more than color accuracy ever will.

**Enforcement:** Before writing `guidance.md` for the next iteration, check the C1 structural audit. If any element is MISSING → the guidance MUST propose a structural fix, not a color/proportion tweak.

## The Loop

### HARD GATE: Iteration Immutability

**CRITICAL: Once `iterations/{N}/output.html` renders successfully in the browser (no JavaScript console errors), that folder is FROZEN.** Any further code change — no matter how small — MUST go in `iterations/{N+1}/output.html`, not as an edit to the existing file.

**Why this matters:** Overwriting a rendered iteration destroys the audit trail. Each iteration is a snapshot we can compare against the reference. If we edit in-place, we lose the ability to track what changed and when, and we cannot measure whether each individual change helped or hurt.

**Enforcement:**
- Before editing ANY `output.html` file, check: has this file already rendered successfully? If yes → create `iterations/{N+1}/` and work there.
- The ONLY exception is fixing JavaScript errors that prevent the file from rendering at all (blank screen, console errors). Once it renders visual output, it is frozen.
- When creating a new iteration folder, COPY the previous `output.html` as the starting point, then make changes to the copy.

### A. Generate

1. Read the current `pattern.json` from the session root (or from the previous iteration's guidance)
2. Apply any corrections from the previous iteration's `guidance.md`
3. **Create the new iteration folder** `iterations/{nn}/` — NEVER reuse an existing folder
4. Save the updated `pattern.json` to `iterations/{nn}/pattern.json`
5. Compile to D3.js HTML following `compile-d3.md`
6. Save as `iterations/{nn}/output.html`

**HTML Requirements** (matching session-abc/test.html format):
- D3.js v7 loaded from CDN: `https://d3js.org/d3.v7.min.js`
- Single `<div id="pattern">` container
- 800x800 SVG canvas (configurable via pattern.json)
- **White body background** (`background: #fff`) — dark backgrounds cause black-screen in Mac Quick Look preview. Apply dark fills to the SVG canvas, not the body.
- Dark fills applied to SVG rect/circle, not body CSS
- Pattern centered in SVG
- NO ES6 template literals (use string concatenation)
- All images embedded as base64 data URIs
- File must be fully self-contained (no external resources except D3 CDN)

### B. Render & Capture (Dual-Image)

Generate TWO screenshots per iteration — **blueprint** (structural edges only) and **final** (fills + edges). This separates topology assessment from visual quality assessment.

1. Open the HTML file in **blueprint mode** first:
   - Navigate to `file:///path/to/output.html?mode=blueprint`
   - Blueprint mode renders edges as thin dark lines on a light background, NO tile fills
2. **Verify render** — check for JavaScript errors immediately:
   - Use `mcp__chrome-devtools__list_console_messages` with `types: ["error"]` to check for JS errors
   - If errors found, use `mcp__chrome-devtools__get_console_message` with the `msgid` for stack traces
   - Fix errors before proceeding — do NOT evaluate a broken render
3. Take blueprint screenshot:
   - Save to `iterations/{nn}/blueprint.png`
   - Read and verify the edge topology looks correct (right number of edges, right connectivity)
4. Navigate to **final mode**:
   - Navigate to `file:///path/to/output.html?mode=final` (or just `output.html` — final is default)
5. Take final screenshot:
   - Save to `iterations/{nn}/screenshot.png`
   - Read and verify it rendered correctly (not blank, not garbled)
   - **This is the primary preview artifact** — Mac Quick Look does NOT execute JavaScript, so `output.html` previews as blank. The `screenshot.png` is the only way to visually preview an iteration from Finder.
6. If output is blank or clearly broken:
   - Check console for errors (step 2)
   - Read the HTML source for obvious bugs (missing variable declarations, syntax errors)
   - Fix and reload — do NOT write an evaluation for broken output
7. **Extract SVG** from the rendered page:
   - Use `mcp__chrome-devtools__evaluate_script` to run: `() => document.querySelector('svg').outerHTML`
   - **IMPORTANT: The evaluate_script result may be wrapped in markdown formatting** (e.g., `# evaluate_script response\nScript ran on page and returned:\n```json\n"<svg..."```). If the result is saved to a file by the tool (when it exceeds token limits), extract the raw SVG from the JSON wrapper using:
     ```python
     import json, re
     with open(tool_result_file) as f:
         data = json.load(f)
     text = data[0]['text']
     match = re.search(r'```json\n(.*?)\n```', text, re.DOTALL)
     svg = json.loads(match.group(1)) if match else text
     ```
   - **Add the SVG namespace** — D3's `outerHTML` produces HTML-flavored SVG without `xmlns`. Add it:
     ```
     sed -i '' 's/<svg /<svg xmlns="http:\/\/www.w3.org\/2000\/svg" /' output.svg
     ```
     Without `xmlns`, browsers render the `.svg` file as HTML text instead of as an image.
   - Save to `iterations/{nn}/output.svg`
8. **Copy reference image** into the iteration folder:
   - Copy `input/reference.{ext}` to `iterations/{nn}/reference.png` (or .jpg) so the iteration is self-contained.
9. **Generate `reference-traced.svg` once per session** (skip if it already exists):
   ```bash
   magick input/reference.jpg -quality 100 /tmp/reference-clean.png
   python3 -c "
   import vtracer
   vtracer.convert_image_to_svg_py(
       '/tmp/reference-clean.png', 'input/reference-traced.svg',
       colormode='color', hierarchical='cutout', filter_speckle=4,
       color_precision=5, layer_difference=15, corner_threshold=60, mode='spline')"
   ```
   Key vtracer params for geometric patterns: `hierarchical='cutout'` (non-overlapping regions), `color_precision=5` (cleaner tile boundaries), `layer_difference=15` (distinct tiles), `filter_speckle=4` (denoise), `mode='spline'` (smooth edges).

10. **Run the validation orchestrator** — single command, single rollup:
    ```bash
    ./tools/iteration-validate.sh \
        --svg iterations/{nn}/output.svg \
        --reference input/reference.jpg \
        --reference-traced input/reference-traced.svg \
        --baseline input/baseline.json \
        --out iterations/{nn}/validation/
    ```
    Pass `--skip-qiyas` if Docker is unavailable. Drop `--baseline` if interpret-pattern hasn't run yet (A6 falls through cleanly).

    The orchestrator runs every signal — `validate-svg.sh`, `qiyas pixel-diff` × 2 (vs traced SVG and vs reference image, via Docker), `qiyas svg-audit` (A1/A2/A4/A5 + A6 when `--baseline` is provided), `qiyas-diff.py`, and `qiyas score` (rollup) — and writes a single `validation.json`. Schema split: `qiyas/docs/validation-envelope.md` (shared envelope) + `docs/validation-overall.md` (sacred-patterns `overall.*` rollup).

    **Read `validation.json` for go/no-go**, drill into `tools/*/` subdirectories for diagnosis:
    - `overall.go_no_go` — `converged` / `iterate` / `broken`
    - `overall.topology_complete` — boolean (validate-svg + A2/A4/A5 all green)
    - `overall.structural_score` — A6 PASS count (e.g. `"5/7"`) or `"n/a"` if no baseline
    - `overall.pixel_similarity` — median of pixel-diff scores (regression detection only, NOT the convergence target)
    - `overall.composite_score` — `qiyas score`'s weighted geometric-mean composite (null when qiyas score didn't run)
    - `overall.warnings[0]` — **the next-action directive**. Ranked by `counterfactual_adjusted_score_delta` (Tax A crystallization × Tax B fragmentation cost) with `counterfactual_score_delta` (raw, upside-only) as fallback — fixing it is expected to deliver the largest *net* score lift after edit cost. `overall.warnings[*].context.counterfactual_rationale` explains *why* this fix raises the score; `counterfactual_tax_rationale` shows the cost adjustment; `counterfactual_fix_archetype` names the construction class the fix belongs to.
    - `overall.blocking_issues` — ranked human-readable list (deterministic gates: validate-svg + A2/A4/A5/A6)

    Per-tool subdirectories under `validation/`:
    - `validate-svg.log` — XML preflight checks
    - `diff-vs-traced/` and `diff-vs-jpg/` — pixel-diff heatmaps and visual overlays
    - `svg-audit/svg-audit.json` — A1/A2/A4/A5 + A6 details (qiyas)
    - `qiyas/report.html` — interactive Tier-3 shape-pairing workbench
    - `qiyas/diff.json` — Hungarian shape pairs and unmatched shapes
    - `score/score.json` — composite + ranked warnings (when `qiyas score` runs)

    Include in `evaluation.md`:
    ```
    ## Validation Rollup (validation.json)
    go_no_go: iterate  |  topology: incomplete  |  structural: 5/7  |  pixel: 74.9%  |  composite: 0.74
    Next action (overall.warnings[0]): missing-shapes — 2 shape(s) in reference have no match in recon (Δ +0.18)
    Blockers: A6 satellite-inner-stars MISSING; A6 band-segments PARTIAL
    ```

11. **Strapwork-overlay diff (when pattern has strapwork)** — re-render with strapwork disabled and run the orchestrator a second time on `output-tiles-only.svg`. The delta isolates tile geometry quality from strapwork rendering quality. If the tiles-only run scores higher than the full run, the strapwork is a net NEGATIVE — fix or remove it before re-adding.

**Why dual images:** The blueprint reveals structural issues (wrong edge count, missing connections, overlapping edges) that are hidden by colored fills. The final reveals visual issues (wrong colors, proportions, white balance). Evaluating BOTH prevents "looks OK at a glance" mistakes.

**Why SVG extraction:** The output.html generates SVG via D3.js at runtime. Extracting and saving the SVG preserves the exact rendered geometry independently of the HTML/JS code. This enables programmatic comparison (tile census, SVG diff) and provides a stable artifact even if D3 CDN changes.

**Chrome DevTools Debugging Tips:**
- `mcp__chrome-devtools__list_console_messages` — list all console output (filter by types: log, error, warn)
- `mcp__chrome-devtools__get_console_message` — get full details including stack trace for a specific message
- `mcp__chrome-devtools__evaluate_script` — run JS in page context for interactive debugging
- `mcp__chrome-devtools__navigate_page` with `type: "reload"` and `ignoreCache: true` — force-reload after fixes
- Add `console.log('Render complete: ' + document.querySelectorAll('polygon').length + ' polygons')` at end of generated HTML for verification

### C. Evaluate

Evaluation has THREE phases, in strict order. C0 is the structured-data primary signal; C1 is reference-grounded human verification of structural elements; C2 is visual-quality scoring. **C0 + C1 must agree before C2 is meaningful.** See Core Tenet T5.

#### C0. Read validation.json warnings (PRIMARY — structured signal)

**Before any human evaluation**, read the orchestrator's structured output. The warnings array carries qiyas's measured assessment of what's wrong and a counterfactual rationale for each issue. C1's reference-grounded comparison verifies and contextualizes C0 — it does not replace it.

```bash
jq '{
  go_no_go: .overall.go_no_go,
  topology: .overall.topology_complete,
  structural: .overall.structural_score,
  pixel: .overall.pixel_similarity,
  composite: .overall.composite_score,
  blockers: .overall.blocking_issues,
  top_warnings: (.overall.warnings // [] | .[0:3])
}' iterations/{nn}/validation/validation.json
```

For each warning, record in `evaluation.md`:

| Field | What it means |
|-------|---------------|
| `id` | Warning kind (`missing-shapes`, `low-geometric-score`, `asymmetric-channel-drift`, `high-param-drift`, `high-position-drift`, etc.) |
| `source` | Which qiyas tool emitted it (`diff`, `pixel-diff`, `zone-audit`, `svg-audit`) |
| `severity` | `error` (gates convergence) / `warn` (degrades score) |
| `message` | Human-readable summary qiyas computed |
| `context.counterfactual_rationale` | "If X were Y, score would improve by Z" — the proposed mechanical fix |
| `counterfactual_score_delta` | **Raw** predicted score lift from fixing this warning (upside-only — does not model edit cost). |
| `counterfactual_adjusted_score_delta` | **Net** predicted lift after Tax A (crystallization probability) and Tax B (fragmentation cost). **This is the actual ranking key — `warnings[0]` is the largest adjusted delta.** |
| `context.counterfactual_fix_archetype` | Construction class the fix belongs to (e.g. `add_strapwork_block`, `tune_existing_parameter`). Determines Tax B's cost basis. |
| `context.counterfactual_tax_rationale` | One line explaining how raw → adjusted (which prior, which fragmentation cost). |

**Why C0 first:** qiyas already did the comparison. Free-form Claude analysis re-derives observations that qiyas measured and ranked. Ground the evaluation in `validation.json` first; use C1 to cross-check that qiyas's top-ranked warning matches what you see in the reference, not to invent a parallel evaluation.

**When C0 and C1 disagree:** if qiyas reports `missing-shapes` but C1 says all elements PRESENT, the disagreement itself is the finding — qiyas is matching against `baseline.json` (user-verified) while C1 is matching against your visual impression. Trust qiyas; revisit `baseline.json` only if you have evidence the user-verified expectations are wrong.

**Record in `evaluation.md`:**

```markdown
## C0. Validation Rollup (validation.json)
go_no_go: iterate  |  topology: incomplete  |  structural: 5/7  |  pixel: 74.2%  |  composite: 0.74

### Top warnings (ranked by counterfactual_adjusted_score_delta — net of fix cost)
1. **missing-shapes** [error, source=diff, Δ +0.172]
   "51 of 139 ref shapes have no match in recon"
   Counterfactual: if recon had the 51 missing ref shape(s), they would match perfectly (param_drift=0)
2. **asymmetric-channel-drift** [warn, source=pixel-diff, Δ +0.052]
   "Channel drift uneven across rotational sectors"
3. **high-param-drift** [warn, source=diff, Δ +0.015]
   "6 pairs with drift >= 0.50"

Blockers: A6 satellite-inner-stars MISSING; A6 band-segments PARTIAL
```

#### C1. Structural Audit (verifies C0 against the reference image)

Open the reference image and the output screenshot side by side. For each architectural element, check: does it EXIST in the output? This is a binary pass/fail, not a quality score.

**CRITICAL: Reference-Grounded Verification**

The structural audit MUST be grounded in the reference image, not in self-defined descriptions. For every element:

1. **Look at the specific zone in the REFERENCE first** — identify what shapes, tiles, and geometry actually appear there
2. **Then look at the same zone in the OUTPUT** — does it match what you see in the reference?
3. **If they don't match, the element is MALFORMED** — even if the output has "something" in that zone

**Anti-pattern (what went wrong for 80+ iterations):** We defined "interstitial bowties" as the element, drew crude dark hexagons, then confirmed "bowties PRESENT." But the reference actually shows partial rosette repetitions with kite and petal tiles — completely different geometry. The audit was checking our own label, not the reference.

**Per-zone visual comparison (mandatory every 5 iterations):**

Every 5th iteration, perform a zone-by-zone visual comparison. Read the reference image and the output screenshot. For each zone below, describe in 1-2 sentences what you SEE in the reference, then what you see in the output, then whether they match:

1. **Center zone** (0-30% of radius): central star polygon
2. **Inner ring** (30-65% of radius): kite tiles
3. **Outer ring** (65-100% of rosette radius): petal tiles
4. **Transition zone** (between central rosette edge and satellite edges): interstitial tiles — what shapes fill this gap? Are they partial rosettes, bowties, kites, or something else?
5. **Satellite interiors**: do the satellite rosettes match the reference's internal structure?
6. **Strapwork bands**: do the white bands follow the same paths and widths?
7. **Band crossings**: is there visible interlacing (over-under)?

### D. Deep Retrospect

**This is the most critical step in the entire process.** The goal is not just to fix what's broken — it's to fundamentally understand what geometric/structural principle you're missing.

#### D0. Walk the validation.json warnings (start here)

Before any open-ended inspection, walk every warning in `overall.warnings` (not just `[0]`) and answer:

1. **Was this warning present in the previous iteration?** Compare `iterations/{nn}/validation/validation.json` to `iterations/{nn-1}/validation/validation.json`. If a warning's `counterfactual_score_delta` shrank, the iteration moved in the right direction. If it grew, the change made things worse.
2. **Did this iteration's edit address `warnings[0]`?** Read the previous iteration's `guidance.md`. If the planned edit was for warning X but warning X's delta did not shrink, the edit failed to land — that's the root cause to investigate, not a new structural theory.
3. **Did a new warning appear?** A regression. The change introduced a problem that was not present before. Identify which edit added it.

This grounds the retrospective in measured deltas before any visual hypothesis-building.

#### The Deep Inspection Protocol

For EVERY difference between your output and the reference (not just MISSING/PARTIAL scores):

1. **Fresh eyes on the reference** — Look at the reference image again as if seeing it for the first time. What do you see NOW that you didn't notice before? Patterns that seemed like details earlier may be fundamental structural elements.

2. **Structural interrogation** — For each difference, ask:
   - "What geometric construction principle produces this shape in the reference?"
   - "Is my analysis.md actually correct, or did I make an assumption that's wrong?"
   - "Am I drawing the right shapes but in the wrong way, or am I drawing the wrong shapes entirely?"
   - "If I overlay my output on the reference, where exactly do the lines diverge?"

3. **Challenge your assumptions** — Don't accept your first explanation. Common blind spots:
   - Assuming the pattern is simpler than it actually is
   - Treating decorative elements as structural ones (or vice versa)
   - Getting the tile topology right but the proportions wrong
   - Missing that what looks like one element is actually composed of multiple overlapping constructions
   - Confusing the strapwork (bands) with the tiles (colored regions between bands)
   - **Curved lines are always circles** — Any curve in the reference is an arc of a circle. Identify which circle (center + radius) and express it as a function of `R`, `n`, `k`. Never approximate with Bezier paths. See SKILL.md "Curved Lines as Circles" for the full protocol.

4. **Root cause, not symptoms** — "The satellites look wrong" is a symptom. "The satellite rosettes in the reference are actually {10/3} stars with filled interstitial kites, not solid dark circles with separation lines" is a root cause. Always dig to the root.

5. **Derive proportions from geometric first principles** — When proportions look wrong, do NOT tweak scaling constants (e.g., changing `R * 0.52` to `R * 0.55`). Instead, derive the correct value from construction geometry:
   - Start from the central circle of radius `R`
   - Inscribe the base polygon ({n} vertices on the circle)
   - Construct the star polygon ({n/k} connections)
   - Compute the mid-ring radius from where adjacent star edges intersect (line-line intersection)
   - Compute satellite distance as: star tip + band width + satellite radius
   - Compute all angles from `360/n` divisions
   - If a measurement cannot be expressed as a trigonometric function of `R`, `n`, and `k`, it is probably wrong

5. **Compare element by element** — Isolate and compare:
   - Central star shape and tile structure
   - Satellite shape, internal structure, and count
   - Interstitial zone topology (how tiles connect star to satellites)
   - Strapwork/band structure (width, continuity, crossings)
   - Color distribution (which zones are light vs dark)
   - Boundary shape

6. **Loop detection** — Before proposing corrections, read ALL previous iteration retrospectives (`iterations/*/retrospective.md`). Check if the proposed approach was already tried and failed:
   - Do NOT repeat the same approach — it will fail again
   - Explicitly note: "Iteration {nn} tried [approach X] which failed because [reason]"
   - Propose a genuinely different approach, or escalate to the user
   - If 3+ approaches have failed for the same element, stop and ask the user for guidance

7. **SVG diff analysis** — Review the diff artifacts produced in Step 10 to ground the retrospective in pixel-level evidence.

   **CRITICAL WARNING — Spatial coverage ≠ structural fidelity:** The SVG diff metrics (similarity %, shared content %, only-in-output %, only-in-reference %) measure **spatial pixel coverage** — whether both images paint similar canvas regions. They do NOT measure whether the **tile shapes, strapwork topology, or interlace structure** match. Two completely different tile topologies can have 95%+ spatial overlap if they cover similar canvas areas with similar colors. ALWAYS perform a manual visual comparison of tile shapes (step 7b below) in addition to reading the diff metrics. Never claim structural match based on diff numbers alone.
   - **`diff/diff-visual.png`** — Red regions = content in our output but NOT in reference (excess). Green regions = content in reference but NOT in our output (missing). Blended = shared. Ask: "Where are the biggest red and green areas? What geometric elements do they correspond to?"
   - **`diff/diff-heatmap.png`** — White = identical, red = different (brighter = larger difference). Identify which regions of the pattern have the highest color or position differences. Ask: "Is the heatmap hottest at tile boundaries (strapwork error), in tile interiors (color error), or across whole regions (structural error)?"
   - **`diff/diff-stats.md`** — Check the convergence metrics:
     - **Overall similarity** trending UP across iterations? If not, the last change made things worse — revert it.
     - **Shared content %** — How much of the canvas has content in BOTH images. Low shared content means scale, position, or structural mismatch.
     - **Only in output %** (red) — Excess content we're drawing that shouldn't be there. Look for: extra tiles, wrong tile shapes extending beyond reference bounds, incorrect fill colors making white areas colored.
     - **Only in reference %** (green) — Missing content. Look for: missing tiles, unfilled regions, shapes that are too small.
     - **Color similarity in overlap** — Where both images have content, how well do colors match? Low color similarity with high structural overlap means the geometry is right but colors are wrong.
   - **Connect diff evidence to error sources** — Each red/green region in `diff/diff-visual.png` should map to a specific error in the error source classification. If you can't explain a diff region, investigate further before proposing corrections.
   - **Trend analysis** — Compare this iteration's `diff/diff-stats.md` against previous iterations. Record the trend in the retrospective. If similarity is DECREASING, the last change was harmful and should be reverted before trying something else.

   **7b. Structural fidelity check** — This is a MANUAL visual comparison that the diff metrics cannot capture. Open the reference and your output side by side and check each of the following. Score each as MATCH / PARTIAL / WRONG:

   | Check | What to compare |
   |-------|----------------|
   | **Tile shape topology** | Do your tile shapes (kites, rhombi, pentagons, hexagons, darts) match the reference? Count distinct tile types in one sector of each. |
   | **Strapwork network** | Do the white bands form the same network pattern? Same number of crossings? Same paths? |
   | **Interlace / weaving** | Does the reference show over-under weaving at band crossings? Does your output? |
   | **Tile-strapwork alignment** | Are your tile boundaries defined BY the strapwork bands (correct) or independent of them (wrong)? In a correct Islamic pattern, tiles are the regions BETWEEN bands. |
   | **Interstitial zone shapes** | In the zone between the central star and satellites, do your tile shapes match? This is often the most complex and most wrong region. |
   | **Satellite internal structure** | Do satellite segments match? Are they bounded by strapwork continuation or just radial wedges? |
   | **Negative space shapes** | Do small dark shapes (darts, stars) at band crossings appear in both? |

   Record the structural fidelity scores in the retrospective.md alongside the diff metrics. If structural fidelity scores are WRONG on 2+ checks, the iteration needs an architectural change, not parametric tuning.

   **7c. Geometric hypothesis testing** — When structural fidelity checks reveal WRONG topology (e.g., wrong tile shapes, missing construction lines, incorrect band count), use systematic geometric investigation:

   1. **Count visible elements in the reference** — How many bands? How many tile types? How many crossings? Use exact counts, not impressions.
   2. **Hypothesize the construction** — Which star polygon ({n/k}) produces the visible bands? Are there additional construction line families? Formulate as a testable hypothesis.
   3. **Test computationally** — Use browser JS (`evaluate_script`) to compute intersections, count faces, and verify the hypothesis produces the expected element counts.
   4. **Compare with reference** — Does the hypothesis produce too many or too few elements? Example: "{10/3} + {10/4} = 20 visible bands, but the reference has only 10 → hypothesis wrong."
   5. **Document in discoveries.md** — Record both confirmed and disproven hypotheses with iteration numbers. This prevents repeating failed approaches.
   6. **Iterate hypotheses** — If disproven, formulate a new hypothesis. Common progression: additional diag families → band-edge intersections → planar face decomposition → connector segments → Hankin method.

   **Key lesson from session 1 (iter 62-63):** Spatial coverage metrics (SVG diff) can show 99%+ overlap even when the tile topology is completely wrong. Two different tile structures can paint similar canvas areas. ALWAYS verify with structural fidelity checks. Never claim structural match based on diff numbers alone.

   **7d. Band-edge intersection investigation** — When tile faces should be bounded by strapwork band edges (not center-lines), use this systematic approach:

   1. **Compute ALL band-edge intersections** — For n diagonals with L/R edges, compute all 20×20 edge-pair intersections. Log total count and radius distribution histogram.
   2. **Set zone boundaries from data, not assumptions** — Examine the actual radius distribution. Set interstitial zone bounds to capture all relevant radii. Don't assume "interstitial starts at r=0.30" — verify with the data.
   3. **Debug per-sector** — For each sector gap, log: which edge pairs produce interstitial points, their exact radii, which bounding edge they lie on (inner S, outer S, inner N, outer N, cross-diag). Compare to expectations.
   4. **Check for degenerate sub-tiles** — If band-edge intersection points are within `halfBandW` of the valley vertex (where center-lines cross), the resulting sub-tiles will be near-degenerate slivers. Merge or skip them.
   5. **Understand the wedge geometry** — The interstitial zone between two non-parallel diags is a wedge that opens from the valley vertex (where they cross at r≈0.289) outward. Band-edge intersections at r≈0.291 are near the apex of this wedge — they subdivide the narrow end, not the wide outer part.
   6. **Classify points by corridor region** — Band-edge intersection points may fall in the band half-tile (C1/C2/C4/C5) or the between-band region (C3). The SC point at r=0.457 falls in C1, not C3. Zone classification must account for which corridor a point belongs to.

#### E2. Mechanically derive next-iteration target from warnings

The next iteration's priority is **`overall.warnings[0]`** (highest counterfactual delta). Translate the warning into a concrete edit using its `id` and `context.counterfactual_rationale`:

| Warning `id` | Mechanical edit derivation |
|---|---|
| `missing-shapes` | Add the missing reference shapes. `context.sample_ref_ids` lists representative ref-shape IDs to start from. Inspect them in `validation/qiyas/report.html` (Tier-3 workbench). Architectural change — never a color/proportion tweak. |
| `A6 ... CLIPPED-MISSING vertex_count=N in zone Z` | A shape the baseline expected at this zone exists *partially* in recon as an open polyline — the construction reaches the right region but is being cut off at the medallion boundary. Fix in bikar DSL: `extend connect every K beyond F` to push the chord endpoints past the boundary, then `clip pattern to medallion_outline` so the partial face becomes a closed clipped shape. The encoder will then re-classify the polyline as a clipped polygon with `params.partial=true`. **Distinct from `MISSING`** — MISSING means the construction wasn't attempted in that zone; CLIPPED-MISSING means it was attempted but didn't reach. See `docs/cross-repo-dependencies.md` §partial-shapes for the qiyas+bikar primitives. |
| `low-geometric-score` | Geometric score < 0.8 means matched shapes have wrong vertex/edge structure. Look at the lowest-scoring pairs in `validation/qiyas/diff.json` and fix per-shape geometry. |
| `high-param-drift` | `context.sample_pair_ids` lists pairs with the worst parameter drift. Compare side-by-side in the Tier-3 workbench; correct the construction parameters (radii, angles) for that shape class. |
| `high-position-drift` | Shapes are roughly correct but mislocated. Recompute positions from first principles (polar coords from center, `360/n` divisions) — likely a magic number leaked into placement. |
| `asymmetric-channel-drift` | Color or fill differs unevenly across rotational sectors. A symmetry violation in coloring — check that color assignment is a function of `(sector_index, ring_index)`, not absolute coordinates. |
| `A2_symmetry: UNEVEN` (svg-audit) | N-fold symmetry broken. Verify `0..n-1` sector-index loop is producing identical sectors. |
| `A5_bands: BROKEN` (svg-audit) | Strapwork band network is incomplete. Check band continuity and crossing count against expected `n*(n-1)/2`. |
| Unknown `id` | Read `context.counterfactual_rationale` verbatim — qiyas already wrote the proposed fix. |

**G2 gate (Architecture Before Pixels) is now mechanical:** if `overall.structural_score` numerator < denominator OR `topology_complete == false`, the next iteration's edit MUST address one of the structural warnings (`missing-shapes`, `A2_symmetry`, `A4_coverage`, `A5_bands`, A6 shape MISSING/PARTIAL). Pixel/proportion edits are permitted only when both topology is complete AND structural is full.

### I. Brutal Honesty Checkpoint (Every 5 Iterations)

**Trigger:** At iterations 5, 10, 15, 20, and every 5th iteration thereafter, STOP the normal incremental loop and perform a full reset evaluation.

The normal iteration loop optimizes within assumptions. Every 5 iterations, those assumptions themselves must be challenged. This is not a deeper version of the Deep Retrospect — it is a fundamentally different mode of thinking.

#### The Protocol

1. **Fresh-eyes comparison** — Open the reference image and your latest screenshot side by side. Pretend you have never seen your output before. Describe what you see in BOTH images as if explaining them to someone else. Do not reference any previous analysis or retrospective.

2. **Ask the hard questions:**
   - "What is FUNDAMENTALLY different between these two images?" — not "what small thing can I tweak?"
   - "What am I getting wrong at the architectural level?" — not "which parameter is off?"
   - "Is my construction method capable of producing the reference, or am I polishing the wrong approach?"
   - "If I were starting from scratch RIGHT NOW, would I build it the same way?"

3. **Challenge the construction method** — The issue may not be parameters, colors, or proportions. It may be that the entire geometric construction strategy is wrong. Consider:
   - Am I decomposing the pattern into the wrong set of primitives?
   - Am I layering elements in the wrong order?
   - Is the tiling topology itself incorrect?
   - Would a completely different algorithm produce this pattern more naturally?

4. **Kill your darlings** — Be willing to throw away the current approach entirely. Sunk cost from previous iterations is not a reason to keep going down the wrong path. If the architecture is wrong, no amount of parameter tuning will fix it.

5. **Write a checkpoint evaluation** in `iterations/{nn}/brutal-honesty-check.md`:
   ```markdown
   # Brutal Honesty Checkpoint — Iteration {nn}

   ## Side-by-side observations (fresh eyes)
   [Describe both images without referencing any prior analysis]

   ## Fundamental differences
   [What is structurally wrong, not cosmetically wrong]

   ## Is the current approach viable?
   [YES: explain why the architecture is sound and only details remain]
   [NO: describe what needs to change and propose an alternative construction]

   ## Decision
   [CONTINUE: stay the course with specific corrections]
   [PIVOT: propose a fundamentally different approach to the user]
   [ABANDON: this construction method cannot produce the reference — escalate]
   ```

6. **Act on the decision** — If the answer is PIVOT or ABANDON, do not continue iterating. Present the findings to the user and propose the new direction before writing any more code.

## Self-Healing Patterns

These patterns reduce iteration count over time by catching common mistakes early:

0. **Validate SVG artifacts immediately** — After extracting an SVG from the browser, run `./tools/validate-svg.sh output.svg` BEFORE proceeding. The script checks: starts with `<svg`, ends with `</svg>`, has `xmlns="http://www.w3.org/2000/svg"`, valid XML, contains visual elements, no wrapper artifacts. If any check fails, re-extract and fix — do NOT continue iterating with broken artifacts.

1. **Verify before evaluating** — Always check console for errors and visually confirm the render is not blank/broken before writing an evaluation. Evaluating broken output wastes an iteration.

2. **Diagnose before retrying** — When output is wrong, read the retrospective from the previous iteration. Understand WHY it was wrong before writing new code. Don't make the same structural mistake twice.

3. **Simplify on failure** — If a complex algorithmic approach (e.g., intersection graph face-tracing) produces wrong results, step back to a simpler approach (e.g., draw known tile shapes directly). Over-engineered solutions are harder to debug.

3b. **Sector-first construction** — For any radially symmetric pattern, build ONE sector (360/n degrees) completely and correctly first, then rotate it n times. Don't try to render each element type independently (central star, then satellites, then tiles). Instead: define every element within a single sector wedge, get that wedge perfect, then let rotation create the full pattern. This guarantees symmetry and makes debugging 10x easier — you only debug one sector. See SKILL.md "Sector-First Construction" for details.

4. **Compare against reference per-element** — Don't evaluate the whole pattern at once. Compare the central star, then the satellites, then the interstitial tiles, then the strapwork separately. Fix the most impactful element first.

5. **Track what changed** — Each iteration's `guidance.md` must clearly state what changed and why. This prevents regression where a fix for one thing breaks another.

6. **Consult learnings first** — Before starting an iteration, read `learnings/common-mistakes.md` for known pitfalls. Many mistakes repeat across sessions.

7. **Root cause before parameter change (Anti-Oscillation Rule)** — Before changing ANY visual parameter (band width, radius, spacing), ask: "WHY does this look wrong?" Classify the root cause:
   - **Architectural**: The rendering MECHANISM is wrong (e.g., polygon strokes double-count shared edges, so apparent band width is 2x what you set). Fix: change the mechanism, not the parameter.
   - **Parametric**: The rendering mechanism is correct, but the VALUE is wrong (e.g., radius too large). Fix: adjust the value, ideally deriving from geometry.
   - **Topological**: The shapes being drawn are the WRONG shapes entirely (e.g., drawing triangles when the reference has kites). Fix: change which shapes are constructed.

   **The oscillation trap**: If you find yourself adjusting a parameter UP in one iteration and DOWN in the next, STOP. You are treating an architectural or topological problem as a parametric one. Step back, identify the mechanism, and fix that instead. Write in the retrospective: "This is an architectural problem, not a parametric one" before proceeding.

   Examples of the oscillation trap:
   - "Make bands thicker" → "Too much white, make bands thinner" → repeat = the problem is HOW bands are drawn (polygon stroke vs explicit lines), not how THICK they are
   - "Make satellites bigger" → "Too crowded, make satellites smaller" → repeat = the problem is spacing/distance formula, not satellite radius alone
   - "Add more interstitial tiles" → "Too cluttered" → "Remove tiles" → repeat = the problem is tile SHAPE/topology, not tile COUNT

8. **Strapwork must match reference geometry or be omitted** — Strapwork (white overlay bands, interlacing) should ONLY be rendered when its geometry correctly matches the reference pattern. Wrong-geometry strapwork (e.g., uniform-width lines along diagonals when the reference has thick woven bands with specific shapes) actively HURTS the metric by placing white pixels at incorrect locations. In Islamic geometric patterns, strapwork bands emerge from the gaps between correctly-shaped tiles — they are not independent overlays. The correct path is: (1) get tile geometry right first (evaluate without strapwork), (2) only add strapwork when it can be geometrically derived from tile boundaries or reference-matched. If the tiles-only metric is higher than the full metric, the strapwork is wrong and should be disabled until it can be corrected.

9. **Express math in terms of pi and n** — Every angle and radius in the pattern comes from `n` (symmetry order), `k` (star polygon skip), and `pi`. Express them that way:
   - Angles: `2*pi/n` (sector), `pi/n` (half-sector), `pi*k/n` (skip angle) — never hardcode `36`, `18`, or `72`
   - Radii: `R * cos(pi*k/n)` (mid-ring), `R * cos(pi*k/n) / cos(pi/n)` (inner) — never use magic decimals like `0.625`
   - Positions: compute from `sector * i` — never from `i * 36 * Math.PI / 180`
   - This makes the geometry self-documenting and portable across symmetry orders. See `declarative-language-guide.md` "Pi-Based Mathematical Conventions" and `compile-d3.md` "Pi-Based Angle Computation" for full reference tables.

### I. Network Connectivity Check

**Rule:** Every geometric element MUST be connected to the unified edge/strapwork network. No floating components.

**When to check:** After every iteration's blueprint render, BEFORE writing the evaluation.

**Detection:**
1. Examine the blueprint render — every drawn edge should visually connect to at least one other edge
2. If any component appears as an isolated shape (e.g., satellite petals disconnected from central star lines), it's a **floating component**
3. A floating component means the element was constructed independently rather than derived from (or connected to) the network

**Fix protocol:**
When a floating component is detected:
1. Identify WHERE it should connect to the network (which intersection point, which line endpoint)
2. Ask: "Is this element DERIVED from the network intersections, or was it constructed independently?"
3. If constructed independently → rebuild it from the network (use intersection points as vertices)
4. If derived but disconnected → check if the connection point coordinates match (floating point rounding issue vs. wrong attachment point)
5. Write in the retrospective: "FLOATING COMPONENT: [element] was not connected to the network at [location]. Fix: [how to connect it]."

**Test scenario:** If the blueprint shows satellite petal edges that don't touch any {10/4} line or intersection point, the check MUST flag: "FLOATING COMPONENT: satellite petals are not connected to the {10/4} edge network. They were constructed independently using satPolarPt() instead of being derived from line intersections."

### I2. Layered Network Decomposition

**Rule:** When a pattern has radial repetition around a point, decompose the network into concentric layers. Each layer's endpoints become the center points for the next layer's radial branches.

**The layered model:**
```
Layer 0: Central midpoint (the origin of the entire pattern)
Layer 1: First radial ring — edges radiating from the central midpoint
         (e.g., star tips, mid-ring vertices)
         These endpoints form a CIRCLE of junction points.
Layer 2: Second radial ring — edges radiating from Layer 1 endpoints
         (e.g., interstitial connectors, satellite approach edges)
         These endpoints are ALSO arranged radially.
Layer 3: Satellite centers — each is a secondary midpoint
         with its own radial sub-network attached
Layer 4: Satellite internal edges — radiating from each satellite center
         (mirroring the structure of Layer 1 at smaller scale)
```

**When to apply:** Every time the blueprint shows radial repetition around ANY point:
1. Identify the center point(s) of the repetition
2. Trace the network outward from each center in concentric rings
3. At each ring, verify that ALL edges at that ring connect to the SAME set of junction points
4. If a layer's edges don't connect to the previous layer's junction points, it's a network attachment error — fix the attachment point, not the edge shape

**Blueprint analysis protocol:**
When examining a blueprint render, mentally decompose it into layers:
1. Mark the central midpoint
2. Draw a circle around the first ring of junction points — are all edges inside this circle correctly radiating from the center?
3. Draw a circle around the second ring — do edges in this ring connect to the first ring's junction points?
4. For each satellite, repeat the same layered analysis (satellite center → satellite edges → satellite boundary)

**Anti-pattern:** Building a satellite's internal edges from `polarPt(satelliteCenter, ...)` without checking that the edge endpoints match Layer 2 junction points. The satellite edges should ATTACH to the Layer 2 ring, not float near it.

**Test scenario:** If a blueprint shows 10 satellites each with 10 petal edges, verify that each satellite's innermost edges share their endpoints with the interstitial connector edges (Layer 2→3 junction). If they don't share endpoints, the satellite is floating despite appearing connected.

### J. Oscillation Detection Check

**Trigger:** Before making ANY parametric change in a new iteration, automatically scan the last 3 iteration retrospectives for oscillation.

**Detection algorithm:**
1. Read the `guidance.md` and `retrospective.md` from iterations N-1, N-2, N-3
2. For each parameter mentioned (band width, satellite radius, spacing, etc.), check if the direction of change reversed:
   - Iter N-2 increased → Iter N-1 decreased (or vice versa) = **OSCILLATION DETECTED**
   - Same parameter adjusted 3+ times in consecutive iterations = **CONVERGENCE STALL**
3. If oscillation detected:
   - STOP — do not make the parametric change
   - Classify the root cause as architectural / topological / parametric
   - Write: "OSCILLATION DETECTED on [parameter]. Root cause is [architectural/topological], not parametric."
   - Propose an architectural or topological fix instead
4. If convergence stall detected:
   - Trigger web research (see Section K)
   - Consider the Brutal Honesty Checkpoint early

**Test scenario:** If iter N-1 increased STRAP from 9→11 and iter N is about to decrease STRAP from 11→9, the check MUST trigger. The response should be: "OSCILLATION DETECTED on STRAP. The issue is architectural (polygon strokes double-count shared edges), not parametric (band width value)."

### M. Scale Calibration Check

**Rule:** At the START of each iteration's evaluation (before scoring individual elements), compare the overall SCALE of the output against the reference. Correct geometry at the wrong scale will look wrong no matter how much you tune details.

**When to check:** Every iteration, as the first step of evaluation (Section C), before scoring symmetry, element count, or proportions.

**Key measurements to compare (output vs reference):**
1. **Medallion outer radius as % of canvas** — Does the overall pattern fill the same proportion of the canvas? If the reference pattern extends to ~85% of the canvas width and yours only reaches ~60%, everything downstream will look "too small" even if the internal geometry is correct.
2. **Star tip radius as % of medallion** — Does the central star occupy the same proportion of the overall medallion?
3. **Satellite center distance as % of medallion** — Are the satellites positioned at the same relative distance from center?
4. **Satellite radius as % of medallion** — Are the satellites the same relative size?

**Method:**
- Mentally overlay the reference and the output at the same canvas size
- Ask: "If I scaled my entire output uniformly, would it match the reference better?" If YES, the base radius R or the canvas scale factor is wrong — fix that BEFORE iterating on details
- Estimate the percentages visually: does the pattern fill the canvas edge-to-edge, or does it float in the middle with large margins?

**Fix protocol:**
When scale is off:
1. Determine whether the output is too large or too small relative to the canvas
2. Adjust the base radius `R` (or the SVG viewBox / canvas dimensions) so the pattern fills the same proportion of the canvas as the reference
3. Re-render and re-check scale BEFORE evaluating any other criteria
4. Do NOT iterate on detail corrections until the scale matches — wrong scale distorts every proportion comparison

**Anti-pattern:** Getting all the internal proportions right (star-to-satellite ratio, band widths, tile angles) but having the whole pattern too small or too large on the canvas. This makes every visual comparison misleading because your eye perceives "wrong proportions" when the real issue is just uniform scaling. Fix scale first, then evaluate proportions.

### Q. Pixel Area Impact Estimation

**Trigger:** Before ANY color correction iteration.

**Purpose:** Color corrections only move the metric when they target regions covering >5% of canvas pixels. Estimate the pixel area before spending an iteration on a color change.
