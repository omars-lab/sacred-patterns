# loop-idle-audit

The operational guard for the single most expensive autonomous-loop failure
mode (retrospective M4): idling — or holding commits, or re-asserting
"converged" — across many ticks when there was unblocked work the whole
time. The bias-for-action tenet (19) and tenet C3 already *say* "idle is not
blocked." This skill is the *check that fires at the moment of temptation*,
which a tenet alone does not.

## When to invoke this skill (the trigger)

Run this **before** you write any of these into a tick's status, a loop
summary, or a Done/Left/Next/Held entry:

- "nothing to do" / "no unblocked work" / "idle this tick"
- "converged" / "we're done" / "the loop is complete"
- "blocked on owner" / "owner-gated" / "waiting for approval"
- "blocked on CI / GHA budget / a push"
- holding N commits without pushing across ≥1 tick

If you're about to assert any of these, you must pass the test below first.

## The idle-vs-blocked test (the whole skill)

Before idling, name exactly ONE of these — out loud, in the tick record:

1. **The specific unfired trigger** you are waiting on — a concrete event
   that has not yet happened (a named CI run completing, a scheduled
   wakeup, an owner reply on a *specific* decision doc by tag). "Waiting for
   work to appear" is not a trigger; name the event.
2. **The named external blocker** — a task#, a decision-doc owner-gate
   (cite the tag + which option is PENDING), or a concrete missing artifact
   (a file that must exist, a baseline that must be regenerated by someone
   else). "I think I'm blocked" is not a blocker; name the dependency.

**If you can name neither, you are NOT blocked — you stopped looking.**
Bias to action on the lowest-tier broken thing (tenet 20): consult the
routing doc (`.claude/plans/post-resurrection-roadmap.md` and the
`qiyas/.claude/plans/post-i1-task-routing.md` §A/§C), `make ledger` for
PENDING decisions you can ship a documented default on (tenet C-decision-
pick), the lowest open task in `TaskList`, or the simplest broken primitive.

### The GHA carve-out (do not mis-read this as a blocker)

A GitHub-Actions budget freeze blocks **pushes**, never **local work**. If
your only "blocker" is "I can't push / CI is frozen / budget exhausted,"
that fails the test — it is not a work blocker. Keep working to the local
tree: mirror CI locally (`make ci-local-fast` in qiyas; the per-repo
simulation targets in CLAUDE.md's GHA-budget tenet), batch docs commits,
accumulate held commits. Holding commits idle across ticks waiting for
budget is the M4 failure (15-tick / 58-commit idle). See memories
[[gha-spend-limit-never-blocks-push]] and [[never-block-on-gha-find-workarounds]].

## What a conforming idle tick looks like

A legitimately-idle tick cites its trigger or blocker explicitly:

> **Next:** idle — waiting on unfired trigger: qiyas#669 owner pick on
> Option D (decision doc PENDING, tag `f3-arrangement-oracle`). No
> lower-tier unblocked work: routing §A clear, ledger has no
> ship-a-default candidates, lowest open task #NN is blockedBy #669.

A **non-conforming** idle tick — the M4 failure this skill prevents:

> **Next:** idle — nothing to do, converged.   ← REJECTED: no trigger, no
> blocker, no proof you checked the lowest-tier work.

  also the `loop-terminal-condition` decision doc for what "done" means.
- **Decision-pick autonomy:** when the test surfaces a PENDING decision doc
  you can ship a documented default on, that is unblocked work — ship it
  (decision-pick authorization), don't re-idle waiting for the owner.
